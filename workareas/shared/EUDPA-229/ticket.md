# EUDPA-229: Process outbox event and publish to SNS Topic

## Metadata
- **Type:** Task
- **Status:** In Dev
- **Priority:** Medium
- **Labels:** pairingCandidate
- **Parent:** EUDPA-125
- **Assignee:** TarunKumar Palisetty

## Description

<p><b>We need to</b> drain the <tt>outbox</tt> collection and publish each event to an SNS topic.<br/>
<b>So that</b> PIMS can react to a notification submission.</p>

<p><b>Background</b><br/>
When a notification is submitted, the backend already writes a domain event to the <tt>outbox</tt> MongoDB collection in the same transaction as the state change (<tt>OutboxService.appendEvent</tt>, event type <tt>NotificationSubmitted</tt>). This is the write half of the transactional outbox pattern.</p>

<p>The publish half does not exist yet. Nothing reads those events or forwards them downstream. This task adds a background poller that publishes each outbox event to an SNS topic, which routes to an SQS queue that the <tt>trade-imports-dynamics-gateway</tt> consumes from and publishes to the ASB so that PIMS can consume notification submissions. SNS publishing is not implemented in the service today — no client, topic, or dependency exists.</p>

<p>The aim here is end-to-end wiring in the TST environment only. We publish to TST just now, the message contract is a first cut, and Judith can clean up any invalid messages in TST. The contract is iterated and hardened in later tickets before we wire up other environments. Nothing is live and nothing is in production.</p>

<p><b>Scope</b></p>
<ul>
	<li><tt>trade-imports-animals-backend</tt> — add the AWS SNS client (SDK dependency + <tt>SnsClient</tt> bean in <tt>AwsConfig</tt>, honouring the LocalStack endpoint override in <tt>AppAwsConfig</tt>).</li>
	<li><tt>trade-imports-animals-backend</tt> — add topic ARN config to <tt>application.yml</tt> / <tt>application-local.yml</tt>, driven by an env var; create the topic in <tt>compose/start-localstack.sh</tt> for local dev.</li>
	<li><tt>trade-imports-animals-backend</tt> — add a <tt>publishedAt</tt> timestamp field to <tt>OutboxEvent</tt> (<tt>null</tt> until published, set to the publish time after a successful SNS publish). <tt>OutboxEvent</tt> has no such field today. A timestamp is preferred over a boolean / status flag because it doubles as audit data — when the event was forwarded downstream.</li>
	<li><tt>trade-imports-animals-backend</tt> — add a scheduled poller (the <tt>@Scheduled</tt> / <tt>@ConditionalOnProperty</tt> pattern from <tt>EmfMetricsPublisher</tt>) that reads un-published events oldest-first and publishes each to SNS. Start from the ADR-EUDP-001 baseline: 2-second poll interval, batch size 10. Make both configurable.</li>
	<li><tt>trade-imports-animals-backend</tt> — set message attributes on each publish (<tt>eventType</tt> plus correlation / schema metadata from <tt>OutboxEventMetadata</tt>).</li>
	<li><tt>trade-imports-animals-backend</tt> — publish under a ShedLock distributed lock so only one app instance polls and publishes at a time (no contention or duplicate publishing across replicas). Follow the prior pattern from <a href="https://eaflood.atlassian.net/browse/EUDPA-168" class="external-link" rel="nofollow noreferrer">EUDPA-168</a>, which already locks the outbox write the same way.</li>
</ul>


<p><ins><b>Acceptance Criteria</b></ins></p>
<ul>
	<li>Given an event with no <tt>publishedAt</tt> in the <tt>outbox</tt> collection, when the poller runs, the event is published to the configured SNS topic and <tt>publishedAt</tt> is then set to the publish time.</li>
	<li>Given an event that already has a <tt>publishedAt</tt>, when the poller runs again, it is not republished.</li>
	<li>Given an SNS publish fails, the event stays un-published and is retried on a later run — no event is lost or skipped.</li>
	<li>Events for the same aggregate are published in ascending <tt>aggregateVersion</tt> order.</li>
	<li>Published messages carry the event payload plus message attributes for <tt>eventType</tt> and correlation / schema metadata.</li>
	<li>When multiple replicas run, only one instance polls and publishes at a time (verified via the ShedLock distributed lock, following the prior pattern from
