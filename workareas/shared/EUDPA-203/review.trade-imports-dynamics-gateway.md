## Summary

Implements the ticket's core feature: a `POST /events` endpoint (JSON-only, 202 on success) that forwards the posted body to an Azure Service Bus queue via `QueueMessageSender` and a SAS-connection-string `ServiceBusSenderClient` built from validated `@ConfigurationProperties`. Failures map to 400 (malformed body), 415 (non-JSON content type), and 502 (ASB send failure) through a new `GlobalExceptionHandler`. Local dev gets a compose-based Service Bus emulator (backed by MSSQL), and integration tests run the full send/receive path against a Testcontainers emulator.

## Positive Observations

- Configuration is exemplary: env-var-sourced keys with empty defaults in `application.yml` plus `@Validated @NotBlank` on the config record give clear fail-fast startup when ASB config is missing — no secrets and nothing TST-specific hard-coded (AC met).
- Error handling has no silent-loss path: send failures are logged with the full exception and surface as 502 with a generic body that leaks no internals.
- The Testcontainers Service Bus emulator setup (`IntegrationBase` + `EventsSendControllerIT`) exercises the real `ServiceBusSenderClient` end-to-end — POST → emulated queue → receive-and-assert — which is unusually strong coverage for a first slice.
- The deleted `ApplicationTest` context-loads smoke test is strictly superseded by the new ITs, which run in CI via failsafe on every PR.
- `QueueMessageSender` is clean: UUID messageId, `application/json` content type, both serialisation and send wrapped into the domain exception.

## Test Coverage

- Unit tests: Present and broad (controller slice, sender, exception handler), but the forwarding-fidelity assertions are weak — no test pins that the published message body equals the posted JSON, and rejection tests don't verify nothing reached ASB (items 13, 14, 16, 21).
- Integration tests: Present and strong in shape (real emulator), with the same fidelity/negative-path gaps plus a PEEK_LOCK isolation hazard (items 21–23).

## Risk Assessment

**Overall Risk:** Medium
**Rationale:** One deployment-blocking concern — the AMQP-over-TCP default transport bypasses the CDP egress proxy (item 7), so the sender likely cannot reach ASB from a deployed environment even though everything passes locally.
