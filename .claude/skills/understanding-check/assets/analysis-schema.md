# Schema: `analysis.{repo}.json`

Per-repo analysis state, one file per repo in scope. Mutated only by:

- `tools/understanding-check/analysis-add-finding.sh` — append a finding.
- `tools/understanding-check/analysis-set-verdict.sh` — mark per-repo
  analysis complete.

**Provenance rule** — every entry in every findings array carries an
`evidence` object pointing at the diff. Helpers reject findings without
evidence. See
[`docs/claude-architect/domain-5-context-reliability/5.6-information-provenance.md`](../../../../docs/claude-architect/domain-5-context-reliability/5.6-information-provenance.md).

## Field reference

| Field | Type | Notes |
|---|---|---|
| `repo` | string | One of the workspace's six repos. |
| `ticket` | string | `EUDPA-XXXXX`. |
| `pr` | number | GitHub PR number for this repo. |
| `commit` | string | Head SHA captured at prepare time. |
| `verdict` | `null` \| `"complete"` | Set by `analysis-set-verdict.sh`. |
| `completed_at` | `null` \| ISO8601 | Set when verdict flips to `complete`. |
| `changeSummary` | `null` \| string (≤300 chars) | One-paragraph summary. |
| `whyItChanged` | `null` \| string (≤300 chars) | Cite ticket goals. |
| `keyDesignDecisions[]` | array | Non-obvious choices. Drop trivial decisions. |
| `edgeCases[]` | array | Inputs at the boundary (large/empty/null/non-ASCII/etc). |
| `failureModes[]` | array | What breaks; how big a blast radius. |
| `securityRisks[]` | array | Anything from the OWASP-flavoured category enum below. |
| `dataOrApiChanges[]` | array | Schema, public API, on-the-wire format. |
| `testCoverageNotes[]` | array | Areas tests do / don't cover. |
| `aiSuspectedRegions[]` | array | Code that looks AI-generated by smell. |

Every nested item has:

- `id` (number, auto-assigned by the helper) — for stable referencing.
- a description field (varies by section, see above).
- `evidence: { file, lines }` — required. Helpers reject if absent.

## Allowed enum values

- `failureModes[].blastRadius`: `local` | `service` | `cross-service`
- `securityRisks[].category`: `injection` | `authz` | `secret` | `pii` | `crypto` | `deps`
- `dataOrApiChanges[].kind`: `schema` | `api` | `wire`
- `testCoverageNotes[].status`: `covered` | `partial` | `missing`
- `aiSuspectedRegions[].why`: `boilerplate` | `stylistic-uniformity` | `over-generic-naming` | `comment-pattern`
- `verdict`: `null` | `complete`

## Scope (categorical, not hedged)

The ANALYST persona is told what to **report** and what to **skip**, not
"be conservative". From
[`docs/claude-architect/domain-4-prompt-engineering/4.1-explicit-criteria.md`](../../../../docs/claude-architect/domain-4-prompt-engineering/4.1-explicit-criteria.md):

**Report**
- Decisions where the diff picked one option over a named alternative.
- Edge-case branches the diff explicitly handles (or notably fails to).
- Failure modes that propagate beyond the immediate function.
- Security risks matching one of the enum categories.
- Schema / API / wire-format changes (any addition or rename).
- Test coverage areas that are demonstrably missing or partial **vs the diff**.
- Code regions that smell AI-generated (boilerplate, over-generic
  naming, uniform comment cadence, no domain idiom).

**Skip**
- Style nits (the `code-style` skill owns those).
- Pre-existing code that the diff did not touch.
- Restating what the diff does. The change summary is one paragraph; the
  rest is gap analysis, not narration.
