# Path-Specific Rules with `.claude/rules` and YAML Frontmatter

This file covers how to scope convention files so they load only when the agent edits matching paths. Consult it when designing a new convention bundle (test-file rules, Terraform rules, language-specific style guides) and you need to choose between a path-scoped `.claude/rules/` file, a subdirectory `CLAUDE.md`, or a single global rule. The goal is to keep irrelevant guidance out of the context window while still covering files that are spread across the codebase.

---

## 1. Use `.claude/rules/` files with a `paths:` frontmatter field for conditional loading

A rule file in `.claude/rules/` is gated by a YAML frontmatter block whose `paths` field is a list of glob patterns. The rule is pulled into context only when the agent is editing a file that matches one of those globs.

This is the mechanism that makes path-specific conventions cheap. Without the frontmatter, every rule you write is global and contends for the same finite context window. With it, a Terraform agent never has to read TypeScript test conventions, and a frontend agent never has to read infrastructure rules.

```yaml
---
paths:
  - "terraform/**/*"
---
# Terraform conventions
- Pin provider versions in `versions.tf`.
- Use `for_each` over `count` for stable addressing.
```

---

## 2. Choose glob patterns that match files by type, regardless of directory

The advantage of path scoping over a subdirectory `CLAUDE.md` is that a glob can target files spread across the tree. Test files, migration files, and generated client stubs typically live next to the code they relate to, not in a single folder — a directory-level `CLAUDE.md` cannot reach them, but a glob like `**/*.test.tsx` can.

Pick the pattern that describes the *kind* of file the rule applies to:

```yaml
---
paths:
  - "**/*.test.tsx"
  - "**/*.test.ts"
---
# Test file conventions
- Co-locate fixtures with the spec that uses them.
- Use `describe` blocks that mirror the public API.
```

A rule keyed to file type stays correct as the codebase reorganises. A rule keyed to a directory does not.

---

## 3. Prefer path-specific rules over subdirectory `CLAUDE.md` for cross-cutting conventions

A subdirectory `CLAUDE.md` is the right tool when a convention truly belongs to one directory tree (e.g., everything under `services/payments/` follows a particular handler pattern). It is the wrong tool when the convention follows a file *type* that recurs throughout the codebase.

The decision rule:

- The convention is bounded by a directory → subdirectory `CLAUDE.md`.
- The convention follows files spread across many directories → `.claude/rules/` with a `paths:` glob.
- The convention is universal → a single project-root `CLAUDE.md` entry.

Mis-applying the first option to a cross-cutting concern forces you to duplicate the same convention in every subdirectory's `CLAUDE.md`, and any drift between copies becomes a silent bug.

---

## 4. Keep each rule file narrowly scoped to reduce irrelevant context

Path-scoped loading only pays off if the rule files themselves are tight. A `.claude/rules/typescript.md` that fires on `**/*.ts` and contains test rules, lint rules, and architecture rules in one file defeats the purpose — the agent now reads everything whenever it touches any TypeScript file.

Split by concern, not by language. One file for test conventions, one for module boundaries, one for API client patterns. Each file's `paths:` block selects the *narrowest* glob that still covers the files the rule applies to.

```yaml
# .claude/rules/test-conventions.md
---
paths:
  - "**/*.test.ts"
  - "**/*.test.tsx"
  - "**/*.spec.ts"
---
```

```yaml
# .claude/rules/api-client.md
---
paths:
  - "src/clients/**/*.ts"
---
```

The agent loads only the rule file whose glob matches the file under edit. Two narrow files cost less context than one broad one.

---

## 5. Treat `paths:` as a coverage contract, not a hint

If a rule must apply to a file, that file's path must match a glob in the rule's frontmatter. There is no fallback — a rule with no matching glob will not load, even if a sibling file in the same directory does.

When adding a new file extension or relocating a directory, audit the `paths:` field of every rule that should still cover it. Missing globs produce silent gaps: the agent edits the file with no awareness of the rule, and the failure looks like a behavioural regression rather than a configuration miss.

For rules where the cost of a miss is high (security policies, public API conventions, migration patterns), prefer slightly broader globs over surgical ones — `**/*.sql` rather than `db/migrations/**/*.sql` — so a new migrations directory does not silently escape coverage.

---

## Workspace verification (EUDPA-275)

This mechanism is confirmed native to Claude Code (official docs, path-specific rules, v2.1.198+): a `.claude/rules/*.md` file with a `paths:` glob frontmatter loads into context when the agent **reads** a file matching any glob in its list — no hook implements it. `tim/.claude/rules/` has relied on this with no supporting hook, and EUDPA-275 added workspace-root rules (`java`, `node`, `gds`, `playwright`, `k6`) on the same basis.

Each rule body only *points at* `docs/best-practices/<topic>/`; it never copies the guidance. Verify a rule is loaded with `/memory` after opening a file that matches its glob.
