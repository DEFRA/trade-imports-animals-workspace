# CLAUDE.md Hierarchy and @import Patterns

This file covers how to organise `CLAUDE.md` configuration across the user, project, and directory levels, how to keep individual files focused via `@import` and `.claude/rules/`, and how to diagnose hierarchy issues when behaviour drifts between team members or sessions. Consult it when designing a new project's configuration layout, when a teammate reports missing instructions, or when a single `CLAUDE.md` has grown large enough that maintainers struggle to find the right section.

---

## 1. Put instructions at the lowest level that owns them

Claude Code reads configuration from three tiers: user-level (`~/.claude/CLAUDE.md`), project-level (`.claude/CLAUDE.md` or root `CLAUDE.md`), and directory-level (subdirectory `CLAUDE.md` files). Each tier has a different audience and a different sharing model.

User-level configuration applies only to the user who wrote it — instructions in `~/.claude/CLAUDE.md` are not shared with teammates via version control. Anything a teammate needs in order to do the work correctly must live in a project-level or directory-level file that is checked in.

Directory-level files let a sub-package or sub-tree carry its own conventions without polluting the root. Use them where the rules genuinely vary by location (a Java service vs a Node frontend vs a tests package), not as a substitute for paragraph headings inside a root file.

---

## 2. Never rely on `~/.claude/CLAUDE.md` for team-wide rules

The most common configuration-hierarchy failure is a new team member not receiving instructions because they live in the maintainer's user-level configuration rather than the project's checked-in configuration. The agent works fine for the original author and silently misbehaves for everyone else.

If an instruction is required for the work to be correct — coding standards, repo layout, branch conventions, deployment rules — it belongs in `.claude/CLAUDE.md` or root `CLAUDE.md`, where it is shared via version control. Reserve `~/.claude/CLAUDE.md` for genuine personal preferences (editor quirks, personal shortcuts) that should never bind teammates.

When auditing a project, walk the user-level file and ask of each rule: "would a new clone of this repo behave correctly without this line?" If the answer is no, move it down into the project tier.

---

## 3. Use `@import` to keep each CLAUDE.md focused on one audience

The `@import` syntax lets one `CLAUDE.md` reference an external file, so a single root or directory-level file can pull in only the standards that are relevant to its package. The maintainer of each package decides — based on their domain knowledge — which standards files matter for that package and imports those, rather than dragging in everything.

This keeps individual `CLAUDE.md` files short and on-topic, and it stops unrelated standards from competing for the agent's attention. A package that has nothing to do with the API layer should not be importing `api-conventions.md`.

```text
# Right — package-level CLAUDE.md imports only what its maintainer needs
@import ../../.claude/rules/testing.md
@import ../../.claude/rules/api-conventions.md

# Wrong — every package imports every standards file, regardless of relevance
@import ../../.claude/rules/testing.md
@import ../../.claude/rules/api-conventions.md
@import ../../.claude/rules/deployment.md
@import ../../.claude/rules/frontend-styles.md
```

---

## 4. Split a monolithic CLAUDE.md into `.claude/rules/` topic files

Once a `CLAUDE.md` grows past a few topics, finding and editing a specific rule becomes harder, and the agent has to load all of it on every session even when only one topic is in play. The remedy is to break the file out into topic-specific files under `.claude/rules/` — for example `testing.md`, `api-conventions.md`, `deployment.md` — and let each `CLAUDE.md` `@import` the subset it needs.

Treat the split as the default once you have three or more distinct topics; treat the monolithic file as a starting shape, not a long-term target.

> For loading a topic by **file type** rather than by `@import`, `.claude/rules/` with a `paths:` glob is the canonical mechanism — see [[domain-3-claude-code/3.3-path-specific-rules]] (verified native, v2.1.198+).

```text
.claude/
  rules/
    testing.md
    api-conventions.md
    deployment.md
CLAUDE.md            # imports the subset relevant at the repo root
packages/
  api/
    CLAUDE.md        # imports testing.md + api-conventions.md
  worker/
    CLAUDE.md        # imports testing.md + deployment.md
```

---

## 5. Verify what is actually loaded with `/memory`

Inconsistent behaviour across sessions usually traces to a memory file that an agent assumes is loaded but is not — a misnamed import, a wrong relative path, or a rule that exists only in the user-level file on one machine. Guessing at the cause from the agent's output wastes time.

Use the `/memory` command to verify which memory files are loaded in the current session. When a teammate reports the agent ignoring a rule, ask them to run `/memory` first; the diff between what they see and what you see will pinpoint the hierarchy issue.

Make this the first diagnostic step for any "the agent isn't following rule X" report, before re-reading the rule or rewriting it more emphatically.

---

## 6. Choose between CLAUDE.md and skills by invocation model

`CLAUDE.md` content is always loaded — every session inherits it. That makes it the right home for universal standards the agent needs available by default, but it also means every line costs context for tasks that have nothing to do with it.

Skills, by contrast, are invoked on demand for task-specific workflows. Move task-specific procedures, verbose checklists, or workflow scripts into a skill rather than expanding `CLAUDE.md`, and reserve `CLAUDE.md` for the rules that genuinely need to bind every session.

When a candidate rule is workflow-shaped ("when doing X, follow steps A, B, C"), prefer a skill. When it is standards-shaped ("we always do X, never Y"), prefer `CLAUDE.md`.
