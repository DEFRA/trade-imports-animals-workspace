# Plan Mode vs Direct Execution

This file covers when to drive Claude Code through plan mode versus direct execution, and how to combine the two within a single task. Consult it when scoping a request — before the first prompt — and again whenever a "simple" change starts surfacing architectural questions. It also covers the Explore subagent, which exists to keep verbose discovery output off the main conversation.

---

## 1. Use plan mode for tasks with architectural implications

Plan mode is designed for complex tasks: large-scale changes, multiple valid approaches, architectural decisions, and multi-file modifications. The point is to explore the codebase and pick a direction *before* committing edits, so you don't pay the cost of rework when investigation reveals the first approach was wrong.

Trigger plan mode when the request has any of these shapes:

- Microservice restructuring or boundary changes.
- Library or framework migrations spanning many files (45+ files is a worked example).
- Choosing between integration approaches with different infrastructure requirements.
- Anything where "which approach" is itself an open question.

The failure mode without plan mode is silent rework — Claude commits to one approach, edits files, and discovery later forces a second pass that contradicts the first. Plan mode collapses that into one pass.

---

## 2. Use direct execution for well-scoped, well-understood changes

Direct execution is appropriate when the change is small, the scope is clear, and there is essentially one correct approach. Two canonical shapes:

- A single-file bug fix with a clear stack trace.
- Adding a single validation check (e.g. a date validation conditional) to one function.

If you can describe the change as a single sentence that names the file and the function, plan mode is overhead. The investigation cost is zero — the only question is implementation, and Claude can implement directly.

The failure mode here is the opposite of rule 1: dropping into plan mode for a one-line fix wastes a round trip and produces a plan document longer than the change itself.

---

## 3. Combine plan mode for investigation with direct execution for the work

Plan mode and direct execution are not mutually exclusive within one task. The intended pattern is:

1. Plan mode to investigate and choose an approach (e.g. plan a library migration).
2. Direct execution to apply the chosen approach across the affected files.

This is the standard shape for migrations and refactors: you need the design discipline of plan mode for "what are we going to do", and the throughput of direct execution for "now do it across 45 files". Treat the transition as an explicit handoff — once the plan is agreed, switch modes rather than continuing to plan-mode each file.

---

## 4. Route verbose discovery through the Explore subagent

The Explore subagent exists to isolate verbose discovery output and return summaries to the main conversation. Use it when investigation will read many files, run many searches, or otherwise produce output that would otherwise consume the main conversation's context window.

The failure mode it prevents is context exhaustion partway through a multi-phase task — the main thread runs out of room because discovery output crowded out the actual work. Push the discovery into Explore, take back a summary, and continue the main task with context intact.

Reach for Explore specifically during the discovery phase of plan-mode work, or any time a single step is going to dump a lot of file contents into the conversation that won't be needed verbatim later.

---

## 5. Match the mode to the task shape, not the developer's preference

The choice between plan mode and direct execution is driven by task shape — scope, number of valid approaches, file count, architectural reach. It is not a per-developer style preference.

Reading a stack trace and patching one conditional is a direct-execution task regardless of who's driving. Restructuring services across repos is a plan-mode task regardless of who's driving. If the team adopts "always plan mode" or "always direct" as a blanket policy, half the time the mode will be wrong for the task in front of it.

When in doubt, look at the two reference points:

- "Single-file bug fix with a clear stack trace" → direct.
- "Library migration affecting 45+ files" → plan.

Place the current task on that axis and pick accordingly.
