# Claude Code in CI/CD Pipelines

This file covers how to invoke Claude Code from CI/CD pipelines so its output is machine-parseable, hang-free, and reusable across re-runs. Consult it when wiring Claude Code into GitHub Actions, GitLab pipelines, or any non-interactive automation — particularly for PR review bots, test-generation jobs, and quality gates that need to post structured findings back to the platform.

---

## 1. Always pass `-p` (or `--print`) for non-interactive runs

Claude Code defaults to an interactive REPL. In CI there is no TTY, so any prompt that expects user input will hang the job until the runner kills it — burning minutes and producing no useful artifact.

The `-p` / `--print` flag forces a single-shot, non-interactive invocation that completes and exits. Make it the first flag on every CI command line, before adding anything else.

```bash
# Wrong — will hang in CI waiting for interactive input
claude "Review the diff in this PR"

# Correct — single-shot, exits cleanly
claude -p "Review the diff in this PR"
```

---

## 2. Enforce machine-parseable output with `--output-format json` and `--json-schema`

Free-text Claude output is fine for humans but brittle for CI: a downstream step that needs to post inline PR comments cannot reliably regex its way through prose. Use `--output-format json` to switch the response into structured JSON, and pair it with `--json-schema` to constrain the shape so the consuming step can rely on specific fields existing.

The schema is the contract between Claude and the rest of the pipeline. When the schema is explicit, the next job (post-to-GitHub, fail-the-build, summarise-to-Slack) can be written against fixed field names rather than guessing.

```bash
claude -p "$REVIEW_PROMPT" \
  --output-format json \
  --json-schema ./ci/review-finding.schema.json \
  > findings.json
```

Downstream steps then iterate `findings.json` and post one inline PR comment per finding without any prose parsing.

---

## 3. Put project context in `CLAUDE.md`, not in the CI prompt

CI prompts are short-lived and rebuilt every run. Stuffing testing standards, fixture conventions, and review criteria into the prompt itself means each pipeline copy drifts from the others, and changes to the standards require touching every workflow file.

`CLAUDE.md` lives in the repo and is picked up automatically. Document there:

- Testing standards (what "good" looks like for unit and integration tests)
- Fixture conventions (where fixtures live, naming, reuse rules)
- Review criteria (what reviewers care about, what they ignore)

The CI prompt then carries only the task ("review this diff", "generate tests for this file"), and the context comes from `CLAUDE.md`. One source of truth, edited by the team, consumed by every pipeline invocation.

---

## 4. Use an independent Claude session to review code another session generated

A session that generated code has the same blind spots when asked to review it: the assumptions baked into generation are the same assumptions baked into the review, so genuine issues go unflagged. This is session context isolation — the reviewing instance must not share state with the generating instance.

Spawn the review as a separate `claude -p` invocation, with its own prompt and no carried-over conversation. Treat generate-then-review as two distinct pipeline steps, even when both run in the same job.

```bash
# Step 1 — generate
claude -p "$GENERATE_PROMPT" --output-format json > generated.json

# Step 2 — review (fresh session, no shared context)
claude -p "$REVIEW_PROMPT" \
  --output-format json \
  --json-schema ./ci/review.schema.json \
  < generated.json > review.json
```

---

## 5. Feed prior review findings back in when re-running a review

When a PR receives a new commit, re-running the review naively will re-flag every issue it found last time — duplicating comments, drowning out new findings, and conditioning reviewers to ignore the bot.

Include the prior findings in the prompt context and instruct Claude to report only new issues or issues that remain unaddressed. The model can then deduplicate against its own previous output rather than the pipeline trying to diff prose.

```bash
claude -p "$REVIEW_PROMPT" \
  --output-format json \
  --json-schema ./ci/review.schema.json \
  --append-system "Prior findings: $(cat prior-findings.json). Report only new or still-unaddressed issues." \
  > findings.json
```

After the run, persist `findings.json` as a workflow artifact so the next push can read it back as `prior-findings.json`.

---

## 6. Include the existing test suite in context when generating tests

A test-generation job that does not know what tests already exist will happily suggest scenarios that are already covered, padding the suite with duplicates and making real coverage gaps harder to spot.

Pass the existing test files (or a summary index of them) into the prompt so Claude can avoid restating what is already tested. Combine this with `CLAUDE.md` documenting which test criteria are valuable — generated output then targets genuine gaps rather than re-asserting things the suite already covers.

The same principle applies to fixtures: documenting available fixtures in `CLAUDE.md` lets generated tests reuse them rather than inventing parallel ones.
