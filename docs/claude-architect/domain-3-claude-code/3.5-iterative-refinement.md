# Iterative Refinement and Progressive Elaboration

This file covers techniques for closing the gap between an under-specified request and a working implementation: concrete examples, test-driven iteration, the interview pattern, and the choice between batched and sequential feedback. Consult it when you are about to send a prose specification that Claude has interpreted inconsistently, when you are starting work in an unfamiliar domain, or when you are about to feed multiple review findings back into a single turn.

---

## 1. Lead with 2-3 concrete input/output examples when prose is being interpreted inconsistently

Natural-language descriptions of a transformation are routinely interpreted differently across runs. The most effective fix is not to rewrite the prose in more careful English — it is to attach concrete worked examples that pin the expected behaviour.

Provide at least two examples (three for any non-trivial transformation), and pick them so they cover the boundaries you actually care about: a typical case, an edge case, and — where relevant — an explicit failure or null case. The examples carry more signal than the surrounding description.

```text
Wrong — prose only
"Normalise the trader reference: strip whitespace, uppercase, and reject anything
that isn't a valid GB number."

Right — prose plus examples
Input:  "  gb 123 456 789  "   →  Output: "GB123456789"
Input:  "gb-123-456-789"        →  Output: "GB123456789"
Input:  "FR123456789"           →  Output: error("non-GB reference")
Input:  null                    →  Output: error("missing reference")
```

When a previous run produced the wrong answer, add the failing case to the example set rather than editing the prose around it.

---

## 2. Write the test suite first, then iterate by sharing failures

For anything beyond a trivial change, ask for the test suite before the implementation — covering expected behaviour, edge cases, and any performance requirements. Then drive the implementation by pasting the failing test output back into the conversation.

This works because the tests turn vague acceptance criteria into a concrete, machine-checkable specification, and the failure output is itself a high-signal worked example: it tells Claude what was expected, what was produced, and where the divergence is. Iteration converges quickly because each turn has a measurable target.

```text
Turn 1 — "Here's the spec. Write a Vitest suite covering happy path,
          empty input, malformed input, and the 10k-row performance budget.
          Do not implement yet."

Turn 2 — "Now implement against that suite."

Turn 3 — "npm test output below. Three failures, all in the malformed-input
          group. Fix them without weakening the assertions."
```

Sharing the raw test output (not a paraphrase) is part of the pattern — the diff between expected and actual is what makes the next turn productive.

---

## 3. Use the interview pattern in unfamiliar domains

When working in a domain you do not fully understand — cache invalidation, distributed failure modes, retry semantics, schema migrations — ask Claude to interview you before implementing. The aim is to surface considerations you would not have thought to specify.

The interview is most valuable for design decisions that are cheap to make up front and expensive to reverse later: invalidation strategy, idempotency requirements, what to do when the upstream times out, how to detect partial writes. A short Q&A round before code is written prevents the larger rework that follows from picking the wrong default silently.

```text
"Before implementing the cache layer, ask me the questions you'd want answered
to design it well. Cover invalidation, staleness tolerance, failure modes, and
how it should behave during a backend outage. Do not write code yet."
```

Use this pattern when the design space is genuinely open. Skip it when the requirement is well understood and you just want the code.

---

## 4. Batch interacting issues; sequence independent ones

When multiple problems surface from a review or a test run, the choice of whether to send them together or one at a time matters.

Send them **together, in a single detailed message** when the fixes interact — for example, a change to a function signature that ripples through three call sites and two tests. Fixing those one at a time produces intermediate states that fail compilation or contradict each other, and each turn re-opens decisions the previous turn already made.

Send them **sequentially** when the issues are independent — for example, a typo in a log message, an unused import, and a missing null check in unrelated files. Sequential iteration keeps each turn focused and the diff readable, and a mistake in one fix does not pollute the others.

```text
Batch (one message):
  - rename `fetchTrader` → `loadTrader`
  - update all 4 call sites
  - update the unit test that asserts the function name
  - update the JSDoc in the public README example

Sequence (one message each):
  - fix the typo in the error string in errors.js
  - remove the unused lodash import in utils.js
  - add a null guard in formatDate in date-utils.js
```

The heuristic is "do the fixes touch each other?". If yes, batch. If no, sequence.

---

## 5. Use specific test cases to fix specific edge cases

When a particular edge case is mishandled — null values in a migration script, a boundary on a date range, a Unicode case-folding subtlety — fix it by providing a concrete test case with the exact input and the exact expected output. Do not describe the edge case in prose and hope the next run handles it.

This is the same pattern as Rule 1, applied surgically to a single defect. The failing case becomes a permanent part of the example set or the test suite, which prevents the same regression on the next iteration.

```text
"The migration drops rows where `decision_date` is null. It should keep them and
set `decision_status = 'PENDING'`. Add a test case:

Input row:   { id: 42, decision_date: null,        decision_status: null }
Expected:    { id: 42, decision_date: null,        decision_status: 'PENDING' }

And a positive control:
Input row:   { id: 43, decision_date: '2025-01-01', decision_status: 'APPROVED' }
Expected:    { id: 43, decision_date: '2025-01-01', decision_status: 'APPROVED' }"
```
