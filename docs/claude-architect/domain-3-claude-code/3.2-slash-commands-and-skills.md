# Custom Slash Commands and Skills

Custom slash commands and skills extend Claude Code with reusable, project- or user-specific workflows. Consult this file when deciding where to define a new capability (project vs personal scope), how to shape its frontmatter, whether it belongs in `.claude/commands/`, `.claude/skills/`, or `CLAUDE.md`, and how to keep its outputs from polluting the main conversation.

---

## 1. Put team-shared commands in `.claude/commands/`, personal ones in `~/.claude/commands/`

Project-scoped slash commands live in `.claude/commands/` and are shared via version control — everyone on the team gets them when they pull. User-scoped commands live in `~/.claude/commands/` and stay on a single developer's machine.

Pick the location by the audience for the behaviour. A command that codifies a team workflow (release prep, ticket scaffolding, lint-and-fix loops) belongs in `.claude/commands/` so it is part of the repository's surface area and evolves under code review. A command tuned to one developer's habits (personal aliases, idiosyncratic prompts) belongs in `~/.claude/commands/` so it does not impose itself on teammates.

Avoid the trap of dropping personal experiments into `.claude/commands/`. Anything committed there becomes a contract teammates rely on; remove it again and you break their muscle memory.

---

## 2. Define skills as `.claude/skills/<name>/SKILL.md` with frontmatter

Skills live in `.claude/skills/` with a `SKILL.md` file per skill. The `SKILL.md` supports frontmatter configuration. The fields you will reach for are:

- `context: fork` — run the skill in an isolated sub-agent context.
- `allowed-tools` — restrict the set of tools the skill may call.
- `argument-hint` — prompt the developer for required parameters when they invoke the skill without arguments.

Treat `SKILL.md` as the skill's contract. The frontmatter declares the operating envelope; the body of the file is the prompt and procedure the model follows when the skill fires.

```markdown
---
context: fork
allowed-tools: [Read, Write, Edit]
argument-hint: "<ticket-id> e.g. EUDPA-1234"
---

# <skill body — instructions, references, examples>
```

---

## 3. Use `context: fork` for skills that produce verbose or exploratory output

`context: fork` runs the skill in an isolated sub-agent context so its outputs do not pollute the main conversation. Reach for it whenever the skill's by-products are not the artefact the parent session needs to keep.

Two clear cases:

- **Verbose output** — codebase analysis, large-file summarisation, log triage. The parent only needs the conclusion, not every intermediate token.
- **Exploratory context** — brainstorming alternatives, scoping options, what-if comparisons. The parent keeps the chosen path; the rejected branches stay forked.

If a skill's output is the answer the parent will act on directly (e.g. a small refactor, a focused edit), a fork is unnecessary overhead — let the skill run inline.

---

## 4. Restrict tool access with `allowed-tools`

`allowed-tools` in the frontmatter limits which tools the skill can call during execution. Use it to make destructive actions impossible rather than merely discouraged.

The principle is least-privilege: a skill that only formats text should not be able to run shell commands; a skill that only writes a new file should not have `Edit` or `Bash`. If the tool is not in `allowed-tools`, the skill cannot reach it regardless of how the prompt drifts.

```markdown
---
# A skill that writes a single new artefact and nothing else
allowed-tools: [Write]
---
```

Pair `allowed-tools` with `context: fork` for skills that explore widely — the fork keeps the noise out of the main session and the allowlist keeps the side-effects bounded.

---

## 5. Use `argument-hint` so invocations without arguments fail loudly

`argument-hint` is the frontmatter field that prompts the developer for required parameters when they invoke the skill without arguments. Set it for any skill that genuinely needs input — a ticket ID, a file path, a target environment.

Without the hint, an argument-less invocation either runs with empty inputs (and produces garbage) or surfaces a confusing error from deep inside the skill body. The hint surfaces the contract at the point of invocation.

```markdown
---
argument-hint: "<EUDPA-XXXX> — the Jira ticket this skill should operate on"
---
```

Keep the hint short and concrete. It is read in a hurry, at the moment of invocation.

---

## 6. Choose skills for on-demand workflows, CLAUDE.md for always-loaded standards

Skills and `CLAUDE.md` are not interchangeable. They differ in when they load:

- **Skills** are invoked on demand for task-specific workflows. They load only when the developer (or the model, via an explicit trigger) calls them.
- **`CLAUDE.md`** is always loaded. Use it for universal standards that should apply to every interaction in the project — coding conventions, repo map, branch-naming rules.

A common mistake is to put a task-specific procedure (e.g. "how to upgrade dependency X") into `CLAUDE.md`. It then occupies context on every conversation regardless of whether it is relevant, and the always-loaded budget bloats. Move task-specific procedures into a skill; reserve `CLAUDE.md` for the standards that genuinely apply universally.

---

## 7. Create personal variants of team skills under a different name

If a team skill in `.claude/skills/<name>/` does not match your personal workflow, create a personal variant in `~/.claude/skills/<different-name>/` rather than editing the shared one.

Editing the project-scoped skill changes the behaviour for every teammate the moment you commit. A personal variant — with a different name — keeps the team's contract intact and lets you iterate on your own preference without coordination cost. Use the project version when collaborating; reach for your personal variant when its differences matter.

The "different name" rule is what makes this safe: same-name personal variants would shadow the team skill and surprise teammates who pair on your machine.
