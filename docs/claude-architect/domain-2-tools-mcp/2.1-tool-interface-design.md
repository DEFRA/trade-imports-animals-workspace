# Tool Interface Design — Best Practices

This file is the authoritative reference for designing tool interfaces that an LLM can route to reliably. Consult it when you are authoring a new tool (skill `tools/`, MCP server endpoint, or in-prompt function), renaming an existing one to disambiguate it from a sibling, or auditing a tool surface that the model keeps misrouting. The rules below all derive from the same root cause: the description and name are the *only* signals the LLM uses to choose between tools, so they must do the disambiguating work explicitly.

---

## 1. Treat the tool description as the model's selection mechanism, not documentation

The description is not human-facing reference material that happens to also be visible to the model. It is the primary mechanism the LLM uses to pick this tool over its neighbours. A terse one-line description leaves the model guessing among similar-sounding alternatives, and the selection becomes unreliable.

Every tool description should explicitly state:

- The tool's purpose in one sentence.
- The expected input format.
- Example queries or invocations that exercise it.
- Edge cases it handles (and ones it doesn't).
- The boundary against neighbouring tools — "use this when X; use `<other_tool>` when Y".

If you skip any of these, the model fills the gap by inference, and the inference is keyword-driven and brittle.

---

## 2. Avoid ambiguous or overlapping descriptions between sibling tools

When two tools have near-identical descriptions — the canonical failure is `analyze_content` versus `analyze_document` — the model misroutes between them. The fix is not to add more prose to both; it is to differentiate them so that the right tool is the *only* plausible match for any given input.

Two levers:

- **Rename** so the name itself signals the boundary. `analyze_content` becomes `extract_web_results` once its description is web-specific. The name now does part of the routing work.
- **Rewrite descriptions** so each one names the input domain, the output shape, and what the other tool does instead.

Right (after rename + rewrite):

```text
extract_web_results: Given a search query string, return the top web results
  (title, URL, snippet) from the public index. Use for queries that should
  hit the internet. For analysing an already-fetched document, use
  summarize_content or extract_data_points.
```

Wrong (the original pair):

```text
analyze_content: Analyses content and returns insights.
analyze_document: Analyses a document and returns insights.
```

---

## 3. Split generic tools into purpose-specific ones with defined input/output contracts

A single generic `analyze_document` collapses several distinct jobs into one signature. The model cannot pick the right behaviour because there is only one tool, and the prompt has to carry all the disambiguation. Output shape varies trip-to-trip.

Split along the natural verbs the caller actually performs:

- `extract_data_points` — pull structured fields from a document.
- `summarize_content` — produce a prose summary of a document.
- `verify_claim_against_source` — check whether a specific claim is supported by a source.

Each one has a tight input/output contract: input is one document plus (where relevant) one question or claim; output shape is fixed per tool. This makes routing deterministic and downstream parsing easier.

The test for whether a split is warranted: if you can describe two distinct output shapes or two distinct expected behaviours for the same tool, it's two tools.

---

## 4. Audit system prompt wording for keyword-sensitive instructions that override tool descriptions

A well-written tool description can still be overridden by stray wording in the system prompt. If the system prompt instructs "always analyse the user's content carefully", the verb `analyse` and the noun `content` create an unintended association with any tool whose name or description contains those tokens — even if a different tool is the correct choice for the request.

When tool selection misbehaves and the descriptions look clean, read the system prompt next. Look for:

- Imperatives that share verbs with tool names (`analyse`, `extract`, `summarise`, `verify`).
- Nouns that share tokens with tool descriptions (`document`, `content`, `source`).
- Instructions that mandate a step ("first, analyse...") which the model then satisfies by picking the most lexically similar tool.

The fix is usually to rephrase the prompt away from the tool's keywords, not to add yet more text to the tool description fighting for primacy.

---

## 5. State "when to use this vs the alternative" inside every overlapping tool's description

Even with good names and a system prompt that doesn't interfere, two tools in adjacent domains will still attract ambiguous inputs. The description itself should resolve the tie.

Each side of a known overlap should name its sibling and state the rule:

```text
summarize_content: Produce a prose summary of an already-fetched document.
  Use when the caller wants a narrative overview. For pulling specific
  structured fields, use extract_data_points instead.

extract_data_points: Pull named structured fields from an already-fetched
  document. Use when the caller has specified the fields they want. For
  a narrative summary, use summarize_content instead.
```

This is redundant prose on purpose. The model reads tools independently when scoring them; the cross-reference inside each description is what makes the boundary visible at selection time.
