# MCP Server Integration

This file covers how to integrate MCP (Model Context Protocol) servers into Claude Code and agent workflows: where to declare them, how to manage credentials, and how to make their tools and resources discoverable. Consult it when adding a new MCP server, deciding between project- and user-scope, writing tool descriptions, or weighing a community server against a custom one.

---

## 1. Scope shared servers at the project, personal servers at the user level

MCP servers can be declared in two places. Use `.mcp.json` at the project root for servers the whole team needs — Jira, GitHub, the team's internal data catalogue. Use `~/.claude.json` for personal or experimental servers that should not be imposed on teammates.

The split matters because project-scoped servers are committed and become part of the repo's contract: every agent that checks the project out gets the same toolset. User-scoped servers stay on the individual machine and don't pollute the shared environment with half-finished experiments.

```jsonc
// .mcp.json — shared, committed to the repo
{
  "mcpServers": {
    "jira": { "command": "...", "args": [...] }
  }
}

// ~/.claude.json — personal, never committed
{
  "mcpServers": {
    "my-scratch-server": { "command": "..." }
  }
}
```

---

## 2. Expand environment variables in `.mcp.json` for credentials

`.mcp.json` is committed. Tokens and secrets must not be. Use `${VAR}` expansion in the file so the structure is shared but the credential resolves from the developer's or CI's environment.

```jsonc
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

Without expansion, you either commit the secret or each developer maintains a divergent local copy of `.mcp.json` — both are failure modes. Expansion keeps the configuration uniform and the credential out of git history.

---

## 3. Assume every configured server's tools are available simultaneously

Tools from all configured MCP servers are discovered when the agent connects and exposed together for the rest of the session. There is no on-demand load; the union is the working surface.

Two consequences follow. First, configuring many servers expands the prompt's tool catalogue, which costs context and can dilute the model's selection. Be deliberate about which servers ship in `.mcp.json` rather than adding "just in case". Second, naming collisions or vague descriptions across servers become visible immediately — design tool names and descriptions assuming peers from other servers will sit next to them.

---

## 4. Write MCP tool descriptions that out-compete built-in tools

Claude Code carries strong built-ins (Grep, Glob, Read, Bash). If an MCP tool's description is thin, the agent falls back to Grep over the codebase even when the MCP tool would answer the question in one call.

Write descriptions that spell out what the tool does, what it returns, and the scenarios it beats a generic search for. State the capability concretely — "returns the full Jira issue including comments, transitions and linked issues as JSON" — not abstractly ("queries Jira").

```jsonc
// Wrong — agent will prefer Grep for anything ambiguous
{
  "name": "get_issue",
  "description": "Get an issue"
}

// Right — agent has enough signal to choose this over Grep
{
  "name": "get_issue",
  "description": "Fetch a Jira issue by key (e.g. PROJ-123). Returns summary, description, status, all comments, transitions, linked issues and parent epic as JSON. Prefer over searching the repo for ticket context."
}
```

---

## 5. Expose content catalogues as MCP resources, not exploratory tool calls

MCP resources are the right shape for catalogue-style data: issue summaries, documentation hierarchies, database schemas. They give the agent visibility into what exists before it commits to fetching anything specific.

Without a resource catalogue, the agent has to probe — listing, sampling, guessing keys — burning tool calls and context. With a catalogue resource, it can scan the inventory in one read and target its follow-up calls.

Treat resources as the "table of contents" layer and tools as the "open the chapter" layer. The same server typically offers both.

---

## 6. Prefer community MCP servers; reserve custom for team-specific workflows

For standard integrations — Jira, GitHub, Slack, common databases — use the existing community MCP server. Maintaining a parallel custom implementation is overhead with no payoff: the community version already handles auth, pagination, schema edges, and updates.

Write a custom MCP server when the workflow is genuinely team-specific: an internal data system, a bespoke ticket lifecycle, a private tool that has no public counterpart. The test is "does anyone outside the team need this?" — if yes, contribute upstream rather than fork; if no, custom is justified.
