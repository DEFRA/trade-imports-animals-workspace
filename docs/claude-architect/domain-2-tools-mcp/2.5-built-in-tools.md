# Built-in Tools — Read, Write, Edit, Bash, Grep, Glob

This file covers how to pick the right built-in tool for a given file-system or
search operation, and how to compose them into an incremental codebase
exploration. Consult it whenever you're designing a skill, a worker prose, or
a CLAUDE.md section that tells an agent how to navigate a repository — and
whenever you're debugging an agent that's reading whole trees up front or
failing to apply edits reliably.

---

## 1. Use Grep for content, Glob for paths

Grep searches the *contents* of files for patterns — function names, error
messages, import statements, string literals. Glob matches files by *path*
shape — naming conventions and extensions like `**/*.test.tsx`. They are not
interchangeable.

Picking the wrong one wastes a turn: `grep` over a path pattern returns nothing
useful, and `glob` over a content string never matches. When the question is
"which files exist that look like X?", reach for Glob. When the question is
"which files mention X?", reach for Grep.

```text
Right — find all callers of a function
  Grep: pattern="processOrder\\(" path="src/"

Right — find every test file under apps/
  Glob: pattern="apps/**/*.test.tsx"

Wrong — Grep for a path pattern
  Grep: pattern="*.test.tsx"
```

---

## 2. Start exploration with Grep, then follow imports with Read

Reading every file in a repo upfront is the slow and expensive way to build
understanding. The fast path is: Grep to find entry points (the route handler,
the CLI command, the failing assertion), then Read those files, then follow
their imports with further Reads.

This keeps context focused on the slice that's actually relevant to the task.
The agent learns the codebase the way a human does — outwards from a known
landmark — rather than trying to memorise the whole map first.

```text
Wrong — Read every file under src/ before answering
Right — Grep "404" -> Read the matching handler -> Read its imports
```

---

## 3. Use Edit for targeted changes; fall back to Read + Write when anchors aren't unique

Edit modifies a file by matching a unique anchor of existing text and replacing
it. It's the right tool when you know exactly which lines to change and the
surrounding text is distinctive enough to identify them unambiguously.

When Edit fails because the anchor text appears more than once — repeated
imports, boilerplate stanzas, duplicate enum values — don't try to make the
anchor longer and longer until it's unique. Switch to Read (to load the full
contents) then Write (to overwrite with the modified version). Read + Write is
the reliable fallback whenever Edit can't find unique anchor text.

```text
Right — single distinctive line, use Edit
  old_string: "const TIMEOUT_MS = 5000;"
  new_string: "const TIMEOUT_MS = 30000;"

Fallback — repeated boilerplate, use Read + Write
  Edit fails because three files import the same symbol on the same line.
  Read the file, modify the string in memory, Write the new content.
```

---

## 4. Trace function usage in two passes: enumerate, then search

When a function is re-exported through a wrapper module, a single Grep for the
original name misses every call site that uses the wrapper. The two-pass
pattern handles this:

1. Read the wrapper module (or Grep for `export` statements in it) to
   enumerate every name it surfaces.
2. Grep across the codebase for each exported name in turn.

Skipping the enumeration step leaves blind spots: refactors miss call sites,
"find all callers" returns a partial list, and downstream changes break in
ways the agent didn't predict.

---

## 5. Match tool granularity to the operation

Read and Write operate on *whole files*. Edit operates on a *targeted region*
inside a file. Choosing the wrong granularity costs either tokens or
reliability:

- Use **Read + Write** when the whole file changes, or when the change is
  hard to express as a unique-anchor replacement (large reformats, multiple
  scattered edits, generated files).
- Use **Edit** when the change is small, local, and the anchor is unique.
- Use **Grep / Glob** to *locate* the file in the first place — don't Read
  files speculatively to find content; Grep them.

The decision tree is: locate with Grep/Glob, inspect with Read, change with
Edit (or Read + Write if Edit can't anchor).
