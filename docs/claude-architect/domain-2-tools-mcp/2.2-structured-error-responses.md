# Structured Error Responses for MCP Tools

This file covers how MCP tools should report failures back to the calling agent so the agent can make sensible recovery decisions. Consult it when designing a new MCP tool, reviewing an existing one's error surface, or deciding what a subagent should propagate to its coordinator versus handle locally. The core principle: errors are data the agent reasons over, not opaque strings.

---

## 1. Use the MCP `isError` flag to signal tool failure

The MCP protocol carries a dedicated `isError` field on tool responses. Return it whenever the tool has failed, even when the body also contains partial results or a human-readable explanation. A successful response with an embedded "error: ..." string is not a failure as far as the agent is concerned — the agent will treat it as a successful result and may proceed as if the operation worked.

Set `isError` first; populate the body second. The flag is what gates the agent's error-recovery branch.

---

## 2. Classify errors into transient, validation, business, and permission categories

A single "Operation failed" string forces the agent to guess at recovery. The agent needs to know whether to retry, reword the input, surface a policy explanation to the user, or escalate for credentials. Those are four different code paths, and the tool is the only component that knows which applies.

Use these categories:

- **Transient** — timeouts, upstream service unavailability, rate limits. Generally retryable.
- **Validation** — invalid input shape, missing required field, malformed identifier. Not retryable without rewriting the call.
- **Business** — policy violation, rule rejection, state-machine refusal. Not retryable; the agent must explain to the user.
- **Permission** — auth missing or insufficient. Not retryable without acquiring credentials.

Return the category as a structured field (e.g. `errorCategory`), not buried in prose.

---

## 3. Return `isRetryable` explicitly — don't make the agent infer it

The agent will otherwise retry on errors that can never succeed (a validation failure, a permanent policy rejection) and waste budget, or skip retries on transient errors that would have worked on the second attempt.

```json
{
  "isError": true,
  "errorCategory": "transient",
  "isRetryable": true,
  "message": "Upstream timed out after 5s; safe to retry."
}
```

```json
{
  "isError": true,
  "errorCategory": "business",
  "isRetryable": false,
  "message": "Consignment cannot be amended after sign-off. Ask the user to raise a new amendment request."
}
```

`isRetryable: false` on business and validation errors is what stops the retry loop.

---

## 4. Pair business-rule failures with a customer-friendly explanation

When a tool rejects a request because of a business rule, the agent is usually going to need to explain the situation to a human. Returning only `isRetryable: false` leaves the agent to invent its own wording — and it may invent something inaccurate or alarming.

Include a plain-language `message` (or equivalent field) that the agent can paraphrase to the user. Phrase it from the user's perspective, not the system's. "Consignments older than 30 days cannot be amended" is usable; "AMEND_WINDOW_EXCEEDED" is not.

---

## 5. Recover transient failures inside the subagent; only propagate what can't be resolved locally

A subagent that hits a transient error should retry it inside its own loop rather than immediately bubbling up. The coordinator's context is a scarce resource — a transient timeout that resolves on retry is noise the coordinator should never see.

When a subagent does have to propagate, send three things up:

1. The structured error (category, retryable flag, message).
2. Any **partial results** already gathered.
3. A short note on **what was attempted** locally (retry count, alternative paths tried).

This lets the coordinator decide whether to dispatch a different subagent, fall back to partial output, or surface the failure to the user — none of which is possible if the subagent just re-raises a bare error.

---

## 6. Distinguish access failures from valid empty results

A query that ran successfully and returned zero matches is a different outcome from a query that could not run at all. Conflating them — for example, returning an empty array in both cases — causes the agent to interpret an outage as "no data exists" and stop investigating.

- **Empty result, successful access:** `isError: false`, empty payload. The agent treats the absence as a real finding.
- **Access failure:** `isError: true`, with category and retryability set. The agent decides whether to retry or escalate.

This split matters most for search, lookup, and list tools, where the "no rows" path is the common case and the failure path is easy to overlook.
