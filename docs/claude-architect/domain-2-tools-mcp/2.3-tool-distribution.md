# Tool Distribution Across Agents and Least-Privilege

This file covers how to scope tool access across multi-agent systems and how
to constrain which tool a subagent calls — and whether it must call one at
all — using the workspace surfaces (SKILL.md, slash-command bodies, subagent
`tools:` allow-lists). Consult it when deciding which tools to expose to a
subagent, when a generic tool should be replaced with a constrained
alternative, or when designing the first step of a workflow that must begin
with a specific tool call.

---

## 1. Scope each subagent's tool set to its role

Giving an agent access to too many tools degrades tool selection reliability —
18 tools versus 4-5 is the canonical comparison. Decision complexity grows
with the number of options, and the model becomes less reliable at picking
the right one.

Restrict each subagent to the tools relevant to its role. A synthesis agent
should not carry a web-search tool just because it might one day need it;
agents-with-tools-outside-their-specialization is a misuse pattern (e.g., a
synthesis agent attempting web searches).

The rule is least-privilege for tools: the smallest set that lets the agent
discharge its role, and nothing more.

---

## 2. Replace generic tools with constrained alternatives

A generic tool surface is also a misuse surface. A worked example is
replacing `fetch_url` with `load_document` that validates document URLs —
the agent still has the capability it needs, but the tool refuses to be
pointed at arbitrary endpoints.

Constraining the tool at the schema level (or in its server-side handler)
moves the validation out of the prompt and into the contract. The agent
cannot misuse what the tool will not accept.

Apply this whenever a generic tool's blast radius is wider than the role
needs.

---

## 3. Provide scoped cross-role tools for high-frequency needs; route complex cases through the coordinator

Strict role-scoping has a failure mode: every cross-role need round-trips
through the coordinator, which is wasteful for high-frequency cases. A
worked example is a `verify_fact` tool given to a synthesis agent — a
narrow, scoped capability that handles the common case in-line.

The trade-off is explicit: scoped cross-role tools for high-frequency needs,
coordinator routing for complex cases. Do not promote a `verify_fact` into a
full research tool just because some facts need deeper investigation; let the
complex cases escalate.

---

## 4. Pin the first tool call via prompt MUST plus a narrow `tools:` allow-list

Some workflows have a mandatory first step — for example, a review skill that
must call `prepare-review.sh` before any per-file reviewer runs. The
workspace has no `tool_choice` knob: the harness, not the skill author,
decides per-turn selection. Two levers do the work instead.

First, state the requirement as a MUST in the SKILL.md body or slash-command
prose. Second, restrict the subagent's `tools:` frontmatter to only the tools
that would satisfy that first step (plus whatever follow-up tools the role
genuinely needs). The narrower the allow-list, the less room the model has
to substitute a near-miss tool.

```yaml
---
name: review-bootstrapper
description: Run the review prepare step before any file-level review fans out.
tools: Bash
---

You MUST call `tools/review/prepare-review.sh EUDPA-X` as your first action.
Do not Read, Grep, or summarise the ticket before that script has run — its
output seeds the workarea every downstream subagent depends on.
```

Use this whenever a downstream step depends on output the agent might
otherwise skip producing. Leaving the first step to "the model will probably
pick it" is the wrong default.

---

## 5. Force a tool call (rather than a prose reply) by restricting `tools:` and stating it in the system prompt

A different failure mode is the subagent replying in prose when the parent
expects a structured artifact — for example, a planner subagent that
"summarises what it would do" instead of writing the plan via the tool that
captures it. Again, the workspace surface is the SKILL.md / slash-command
body plus the subagent's `tools:` list, not an API parameter.

Restrict `tools:` so the only useful action is the tool you want called, and
state in the system prompt that the subagent must call that tool before
returning. If the subagent has no Read/Grep/etc. to fall back on, prose
becomes the obviously wrong shape and the model picks the tool.

```yaml
---
name: plan-writer
description: Persist a plan via the planning tool — no prose replies.
tools: mcp__planning__write_plan
---

You MUST call `mcp__planning__write_plan` before returning. Do not summarise
your intent in prose — the parent agent reads the tool result, not your
reply.
```

This pairs with rule 4: rule 4 pins which tool, rule 5 guarantees some tool
is called. Use rule 5 when conversational text is the wrong output shape and
any of the allow-listed tools is acceptable.

---

## 6. Treat tool-count as a reliability signal, not a feature count

The 18-versus-4-5 comparison is a quantitative claim: tool selection
reliability degrades as the available set grows. When an agent's
tool list creeps past a handful, that is a design signal to split the agent,
push tools behind a coordinator, or collapse overlapping tools into one.

Right:

- Synthesis agent: `load_document`, `verify_fact`, `cite_source` (3 tools).
- Research agent: `web_search`, `load_document`, `summarise` (3 tools).

Wrong:

- One agent: 18 tools spanning search, synthesis, verification, citation,
  storage, and notification.

If a role genuinely needs many tools, that is evidence the role should be
decomposed, not that the model should be asked to choose harder.
