# Claude Code Hooks for Tool Call Interception

In Claude Code, hooks are configured in a `settings.json` file (user, project, or directory scope) under the `"hooks"` key. The hook event names are: `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Notification`, `Stop`, `SubagentStop`, `PreCompact`, and `SessionStart`. Each maps to a list of matchers plus shell commands that the harness executes when the event fires. Hooks have full access to the tool's input and output, can block the call (`PreToolUse` exit-code 2), and can inject feedback the model sees (`PostToolUse` stdout).

Use hooks for guardrails that prompt-engineering cannot enforce reliably: formatting (run prettier after every `Write`), security gates (refuse `npm install` of unscoped packages), policy enforcement (block `git push --force` on `main`), audit trails (log every `Bash` command run). Consult this file when designing such a guardrail, or when a business rule (refund cap, policy gate, escalation path) must hold deterministically rather than rely on the model following its instructions.

*The Agent SDK has equivalent semantics for the same event names — if you ever stand up an SDK-based agent, the principles translate directly.*

---

## 1. Use PostToolUse hooks to normalise heterogeneous tool output

Different MCP tools return the same conceptual data in incompatible shapes — one tool emits Unix timestamps, another ISO 8601 strings, a third numeric status codes. Asking the model to reconcile these on the fly burns context, invites silent mis-parsing, and makes downstream behaviour depend on which tool happened to be called.

Configure a `PostToolUse` hook matched to the relevant tool name (or MCP server prefix) that pipes the tool result through a normalising script. Anything the script writes to stdout is appended to the tool result the model sees, so the model reasons over one canonical shape rather than re-deriving the same normalisation each turn.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "mcp__.*__get_.*",
        "hooks": [
          { "type": "command", "command": "tools/normalise-tool-output.sh" }
        ]
      }
    ]
  }
}
```

Use this pattern when the agent calls more than one tool that returns the same logical field (timestamps, statuses, identifiers) in different formats.

---

## 2. Use PreToolUse hooks to enforce hard policy limits

Some rules must hold every time — writes to a restricted resource, destructive git operations on `main`, `npm install` of an unscoped package, actions outside an approved scope. Embedding the rule in `CLAUDE.md` or a skill prompt makes compliance probabilistic: the model will usually obey, but "usually" is not a guarantee.

Intercept the outgoing tool call with a `PreToolUse` hook and exit non-zero (exit code `2`) when the arguments violate the rule. The harness aborts the tool call before it runs, so a `Bash` call containing `git push --force origin main` cannot reach the shell regardless of how the model was prompted or jailbroken. Stderr from the hook is surfaced to the model as the reason for the block.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "tools/guard-destructive-git.sh" }
        ]
      }
    ]
  }
}
```

Pair the block with a redirect to an alternative workflow — a safer command form, a request-approval step, a narrower-scope variant — so the agent has somewhere to go rather than dead-ending.

---

## 3. Choose hooks over prompts when compliance must be guaranteed

`CLAUDE.md` / skill instructions are probabilistic enforcement. Hooks are deterministic enforcement. Pick the mechanism that matches the consequence of failure.

| Rule type | Mechanism |
|-----------|-----------|
| Style, tone, preferred phrasing | `CLAUDE.md` / skill prompt |
| Default behaviour the model should usually follow | `CLAUDE.md` / skill prompt |
| Hard financial / regulatory / safety limit | Hook |
| Formatting that must run after every edit | Hook |
| Anything where a single violation is unacceptable | Hook |

If you find yourself writing "the model MUST NOT" in capital letters in `CLAUDE.md`, that is the rule's way of telling you it belongs in a hook.

---

## 4. Redirect blocked actions, do not just refuse them

A hook that blocks a tool call without offering an alternative leaves the agent stuck — it will retry, rephrase, or escalate in unhelpful ways. When a `PreToolUse` hook intercepts a policy-violating action, the stderr message it emits is the only signal the model has about what to do next. Use that channel deliberately: name the legitimate path (the approval workflow, the scoped command form, the human-escalation skill) rather than just naming the violation.

This converts the hook from a wall into a routing layer. The agent learns from the hook's response that the path it tried is closed and where the open path is, without `CLAUDE.md` needing to enumerate every edge case.

---

## 5. Keep hook logic narrow and side-effect free where possible

Hooks run on every matching tool invocation and sit in the hot path of Claude Code's agent loop. A `PostToolUse` hook that does heavy I/O on every call slows every turn; a hook with hidden side effects makes the harness's behaviour hard to reason about.

Limit hooks to:

- Pure data transformation (timestamp normalisation, field renaming, unit conversion).
- Argument inspection and policy decisions (allow / block via exit code).
- Deterministic post-processing tied to the matched tool (formatter on `Write`, linter on `Edit`).
- Light logging or audit emission tied to the compliance purpose of the hook.

Heavier work — enrichment that requires another API call, multi-step business logic — belongs in a dedicated skill or MCP tool the agent can choose to call, not in a hook that fires unconditionally on every matching tool invocation.
