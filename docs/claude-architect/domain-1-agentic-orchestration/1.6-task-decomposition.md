# Task Decomposition Strategies for Complex Workflows

This file covers how to break a complex agentic workflow into smaller, tractable subtasks: when to fix the steps up front (prompt chaining) versus when to let the plan evolve as findings come in (dynamic decomposition). Consult it when designing a multi-step skill, a review pipeline, an investigation harness, or any workflow large enough that a single prompt would lose focus.

The core trade-off is **predictability vs. adaptability**: chained pipelines are easy to reason about and reproduce, dynamic plans are needed when the shape of the work is only knowable after the first pass.

---

## 1. Use prompt chaining when the workflow is predictable and multi-aspect

If you already know every step the workflow needs — and the steps don't change based on what the model finds — model the workflow as a fixed sequential pipeline. Each step gets a focused prompt, with the previous step's output flowing in as context.

This is the right shape for predictable multi-aspect reviews, where the aspects are enumerable up front: lint, then type-check, then test-coverage, then security. Each aspect benefits from a dedicated prompt with its own rules; a single mega-prompt covering all four would dilute attention and produce thinner findings on each axis.

The chain also gives you natural checkpoints — you can inspect, gate, or short-circuit between steps without re-running the upstream work.

```text
Step 1 — analyse file A
Step 2 — analyse file B
Step 3 — analyse file C
Step 4 — cross-file integration pass over outputs of 1-3
```

---

## 2. Use dynamic decomposition when the workflow is open-ended

When the shape of the work depends on what you discover — "add comprehensive tests to a legacy codebase", "investigate why this incident occurred", "modernise this service" — a fixed pipeline cannot enumerate the steps in advance. Use dynamic adaptive decomposition: generate the next subtask from what the previous step uncovered.

The pattern for open-ended investigation tasks is:

1. **Map structure** — first pass establishes the lay of the land (modules, entry points, dependency graph, hot spots).
2. **Identify high-impact areas** — second pass picks where to focus based on (1).
3. **Build a prioritised plan that adapts** — as you work each area, new dependencies surface and the plan reorders or branches.

Don't try to write the full plan up front when the inputs to the plan don't yet exist. Write the next two or three steps, execute, and let the findings drive what comes next.

---

## 3. Split large code reviews into per-file passes plus a cross-file integration pass

Loading every file in a review into a single prompt dilutes attention — the model spreads its scrutiny too thinly across files and misses local issues that a focused per-file pass would catch.

The correct decomposition is two layers:

- **Per-file local analysis** — one prompt per file, focused on that file's own logic, style, and correctness. Run these in parallel where the orchestration layer supports it.
- **Cross-file integration pass** — a separate, later step that consumes the per-file outputs and looks for issues that only exist at the seams: inconsistent naming, contract drift between caller and callee, duplicated logic, ordering or timing assumptions.

Both passes are necessary. The per-file pass catches everything local; the integration pass catches everything global. Folding them into one prompt loses both.

```text
Wrong:
  one prompt receives all 12 files and is asked to "review the change"

Right:
  12 parallel per-file prompts → 1 integration prompt that reads the 12 outputs
```

---

## 4. Match the decomposition pattern to the workflow's predictability

Don't default to one strategy. The decision is:

| Workflow trait                                  | Decomposition |
|--------------------------------------------------|---------------|
| Steps known in advance, aspects enumerable       | Prompt chaining (fixed pipeline) |
| Steps depend on findings, scope only knowable mid-flight | Dynamic adaptive decomposition |
| Large surface, parallelisable per-unit work      | Fan-out per unit + dedicated integration pass |

Picking the wrong pattern is expensive: a chained pipeline applied to an open-ended task forces premature commitment to a plan that won't survive contact with the codebase, while a dynamic plan applied to a predictable workflow burns tokens re-deciding what should have been a static sequence.

---

## 5. For legacy-codebase tasks, map structure before you plan work

When asked to do something broad to an unfamiliar codebase ("add comprehensive tests", "audit for security issues", "modernise dependency usage"), do not jump straight to executing the task. The first subtask is always to map the structure: what modules exist, how they depend on each other, where the high-impact areas are.

Only once that map exists can you build a prioritised plan. And that plan must be allowed to adapt — as you work on one area, you will discover dependencies on others that reorder the queue. Treat the plan as a living artifact, not a contract.

The failure mode of skipping the mapping step is that you produce uniform, low-value output (e.g. tests for the easiest files, not the highest-impact ones) and then run out of context before reaching the work that mattered.
