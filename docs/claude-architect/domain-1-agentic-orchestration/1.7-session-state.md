# Session State, Resumption, and Forking

In Claude Code, sessions are persisted per-project on disk. Pick up an existing session with `claude --resume <session-id>` or the interactive `/resume` command; list available sessions with `claude --resume` (no id); fork a session with `claude --resume <id> --fork-session` to branch a new investigation without disturbing the original. Sessions can be named at launch (`claude --session-name "feature-x"`) so you can find them later. Each session carries its own messages array, working directory, and tool permissions snapshot.

Covers how to manage Claude Code session state across work sessions: when to resume a named session, when to fork one, when to abandon a session and start fresh with a structured summary, and how to keep a resumed agent honest about file changes it didn't witness. Consult this file when designing skills or workflows that span more than a single agent invocation — anything that talks about "picking up where we left off", branching an analysis, or comparing two approaches from a shared baseline.

*The Agent SDK exposes the same forking/resume primitives programmatically (named sessions, `fork_session` option) — the conceptual model is identical.*

---

## 1. Name sessions you intend to resume; use `--resume <session-name>` to return to them

Anonymous sessions are disposable. The moment a session is worth continuing later — a long investigation, a multi-step refactor, a research thread — give it a name at launch so you can address it again with `--resume`.

Without a name, "resuming" degrades into scrolling through `claude --resume` (no id) trying to spot the right session in the list, or copy-pasting context back in — exactly the failure mode session resumption is designed to avoid. With a name, the same prior conversation continues with its tool results and reasoning intact.

```bash
# Launch with a stable name
claude --session-name "investigate-flaky-auth-tests"

# Days later, return to it
claude --resume investigate-flaky-auth-tests
```

Treat the session name as a stable identifier — pick something descriptive enough that you, or another agent, can find it again days later. If you didn't name it at launch, run `claude --resume` with no argument to pick the session from the interactive list before the right one rolls off the history.

---

## 2. Use `--fork-session` to explore divergent approaches from one shared baseline

When a single analysis legitimately branches — "should we refactor by extracting a service, or by inverting the dependency?" — don't run two independent sessions that each have to redo the codebase analysis. Fork the session.

```bash
# Branch a new investigation from a shared analysis baseline
claude --resume investigate-flaky-auth-tests --fork-session
```

`--fork-session` creates an independent branch from the parent session's accumulated context. Each fork explores its approach without polluting the other, and both inherit the same prior tool results and reasoning for free. This is the right primitive for A/B comparisons of testing strategies, refactoring approaches, or design options that all start from the same understanding of the code.

A new session for each branch wastes the shared exploration. A single linear session that tries to hold both branches at once entangles them. Forking is the structural fit.

---

## 3. Inform a resumed session about files that changed while it was paused

A resumed session believes its prior tool results reflect current reality. If the codebase moved on between sessions — edits, refactors, dependency bumps, a `git pull`, a subagent's commits — the agent will reason from stale `Read` snapshots and produce confidently wrong output.

Before asking a resumed session to act, tell it which files have changed so it can re-read those specifically. Targeted re-analysis is faster and cheaper than forcing a full re-exploration, and far safer than letting the agent run on stale context.

```text
# Right — tell the resumed agent what moved
Resuming. Since our last session, these files changed:
  - src/auth/session.ts (refactored to use the new TokenStore)
  - src/auth/session.test.ts (tests updated to match)
Please re-read those before continuing the investigation.

# Wrong — assume the prior view still holds
Resuming. Continue where you left off.
```

The principle: prior tool results are evidence with a timestamp. When the underlying files have moved past that timestamp, the evidence is invalidated and the agent must be told. A `SessionStart` hook in `settings.json` can automate the nudge:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "resume",
        "hooks": [
          { "type": "command", "command": "git diff --name-only @{1.day.ago} 2>/dev/null | head -50" }
        ]
      }
    ]
  }
}
```

The hook's stdout is injected into the resumed session's context, so the agent sees the changed-files list before its first turn.

## 4. Start a new session with a structured summary when prior tool results are stale

Resumption is only the right choice when prior context is mostly still valid. If too much has changed — large refactor, dependency rewrite, a long gap — resuming with stale tool results is less reliable than starting fresh with a well-written summary injected at the top.

A structured summary lets the new session rebuild context deliberately, against the current codebase, with no half-true cached observations to mislead it. Resumption is faster when it's safe; a fresh session with a summary is more reliable when it isn't. Pick based on how much of the prior context survives the gap, not on which is easier to type.

| Prior context status | Choose |
|----------------------|--------|
| Mostly valid; small targeted changes | `claude --resume <name>` plus an "informed about changes" message (§3) |
| Largely stale; significant churn since the session was paused | New session seeded with a structured summary |

In Claude Code, the structured-summary path is usually: run `/compact` near the end of the prior session to produce a clean handoff, save the summary alongside the workarea, and paste it as the first turn of a fresh `claude` invocation.

---

## 5. Choose between resumption and a fresh-with-summary on evidence, not habit

The decision is not stylistic. It's a judgement about whether the prior session's tool results — `Read` results, `Grep` hits, `Bash` outputs — still describe the system the agent will act on.

Ask, before resuming: how many of the files this session touched have changed? How many of its assumptions still hold? If the answer is "almost all still valid", resume and inform about the diff. If the answer is "I'd have to caveat half of it", start fresh with a summary. Defaulting to `--resume` because it's the named convenience flag is the failure mode this rule exists to prevent.
