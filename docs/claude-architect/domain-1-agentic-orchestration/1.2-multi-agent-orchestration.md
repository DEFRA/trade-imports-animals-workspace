# Multi-Agent Orchestration — Coordinator/Subagent Patterns

This file covers the hub-and-spoke pattern for multi-agent systems: how a coordinator decomposes a task, selects and invokes subagents, partitions scope between them, and iteratively refines the synthesis. Consult it when designing any skill, workflow, or MCP-fronted system that fans work out to more than one Claude invocation and then aggregates the results.

It pairs with [[domain-1-agentic-orchestration/1.3-subagent-invocation]] (the mechanics of `Task`, context passing, and `AgentDefinition`); this file focuses on the architectural pattern, not the API surface.

---

## 1. Route all subagent traffic through the coordinator (hub-and-spoke)

In a hub-and-spoke architecture the coordinator owns every inter-subagent message, every error, and every information hand-off. Subagents do not call each other. They report back to the coordinator, and the coordinator decides what (if anything) the next subagent sees.

Centralising the traffic is what gives you observability, consistent error handling, and controlled information flow. A peer-to-peer topology where subagents talk to each other gives up all three: failures surface inconsistently, the trace fragments across agents, and a single misbehaving subagent can poison the context of every downstream caller.

```
Wrong — peer-to-peer       Right — hub-and-spoke
   A → B → C                   A ← coordinator → B
        ↘ D                              ↓
                                         C, D
```

If you find yourself wanting subagent B to "just call" subagent C, that is a signal the coordinator's plan is incomplete — fix the plan, don't bypass the hub.

---

## 2. Treat subagent context as isolated by default

Subagents do not inherit the coordinator's conversation history. They do not share memory across invocations. Every spawn starts from a clean slate with whatever prompt the coordinator hands it.

This is a feature, not a bug — it keeps subagent reasoning focused and prevents context bleed — but it has a hard consequence: the coordinator must pack everything the subagent needs into the prompt itself. Prior search results, document excerpts, attribution metadata, quality criteria: if it isn't in the prompt, it isn't available.

When debugging a "the subagent didn't know X" failure, the first question is always: did the coordinator put X in the spawn prompt? Usually it didn't.

---

## 3. Make the coordinator do task decomposition, delegation, aggregation, and routing decisions

The coordinator has four distinct jobs and you should be able to point to each of them in the system prompt:

1. **Decompose** — break the user's query into subtasks.
2. **Delegate** — decide which subagent type handles each subtask.
3. **Aggregate** — combine the subagents' returned findings into a single answer.
4. **Route by complexity** — decide *which* subagents to invoke based on the query, rather than always firing the full pipeline.

The fourth job is the one most often forgotten. A coordinator that always runs `search → analyse → synthesise → critique` for every query wastes tokens on simple lookups and underserves complex research questions that need extra refinement passes. Design the coordinator to analyse query requirements first and dynamically select the subagent set.

---

## 4. Watch for overly narrow decomposition on broad topics

The most common coordinator failure on research-style tasks is decomposing the query too narrowly — picking three crisp subtopics that leave half the question uncovered. The synthesis looks coherent, but a reader who knows the domain spots the gap immediately.

Mitigate this by:

- Including breadth as an explicit quality criterion in the coordinator prompt ("identify all major subtopics, not only the obvious ones").
- Having the coordinator enumerate candidate subtopics *before* committing to which ones to delegate.
- Treating the first synthesis as a draft, not the final answer (see rule 6 on iterative refinement).

A coordinator that confidently delegates three narrow tasks on a broad question will produce a confidently narrow answer.

---

## 5. Partition scope across subagents to minimise duplication

When you fan out to parallel subagents, give each one a distinct slice of the work — a different subtopic, a different source type, a different document set. Two subagents pointed at the same scope will return overlapping findings, double the token cost, and force the coordinator to deduplicate during aggregation.

Concrete partitioning strategies:

- **Subtopic split** — agent A covers regulatory context, agent B covers technical implementation, agent C covers operational risk.
- **Source-type split** — agent A searches academic papers, agent B searches news, agent C searches internal documents.
- **Time-window split** — agent A covers the last 12 months, agent B covers historical context.

The coordinator's delegation prompts should make the boundary explicit so each subagent knows what is *not* its responsibility.

---

## 6. Implement iterative refinement loops, not single-pass pipelines

A single pass — decompose, delegate, synthesise, return — is rarely enough for non-trivial research. Build the coordinator to evaluate its own synthesis output, identify gaps, re-delegate to search and analysis subagents with *targeted* follow-up queries, and re-invoke synthesis until coverage is sufficient.

The loop:

1. Synthesise the current findings.
2. Critique the synthesis: what's missing, what's thin, what's contradicted.
3. If gaps exist, spawn targeted subagents to fill them (not a full re-run of the pipeline).
4. Re-synthesise. Stop when coverage criteria are met or an iteration budget is hit.

The discipline that makes this work is the targeted re-delegation. Replaying the original decomposition is wasteful; the second pass should look nothing like the first because the gaps it's filling are specific.

---

## 7. Specify goals and quality criteria, not step-by-step procedures

Coordinator prompts that read like a flowchart ("first do X, then do Y, then do Z") constrain subagents to the coordinator's pre-baked plan and lose the adaptability that makes the multi-agent setup worth the cost.

Instead, write the coordinator (and subagent) prompts in terms of research goals and quality criteria. State what "good" looks like — coverage, attribution, source diversity, depth — and let the agents adapt their approach to the specific query.

```
Wrong — procedural
  "First call web_search with the query. Then call fetch_url on the
   top 3 results. Then summarise each. Then write the answer."

Right — goal-and-criteria
  "Produce a research brief that covers regulatory, technical, and
   operational angles. Cite primary sources where possible. Flag
   any claim that rests on a single source. Stop when each angle
   has at least two independent citations."
```

The procedural version gives you the same shape of answer every time, even when the query doesn't suit it. The goal-and-criteria version lets the coordinator pick a different decomposition for a different question.
