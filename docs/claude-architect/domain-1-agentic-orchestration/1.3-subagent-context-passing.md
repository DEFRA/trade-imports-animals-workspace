# Subagent Invocation, Context Passing, and Isolation

This file covers how a coordinator agent spawns subagents, what state crosses the boundary, and how to keep findings attributable end-to-end. Consult it when designing a fan-out workflow, a `.claude/agents/<name>.md` subagent definition, or any prompt that hands work from one agent to another. The core principle is that subagents are isolated processes — nothing travels between them except what you explicitly write into the spawning prompt.

---

## 1. Allowlist `Task` on the coordinator, not the workers

The `Task` tool is the only mechanism for spawning a subagent. A coordinator that is supposed to fan work out must have `"Task"` in its `allowedTools`, otherwise the orchestration pattern silently degrades to single-agent execution.

Workers, by contrast, should not normally carry `Task` — that would let leaves spawn their own leaves and turn a tree into an unbounded graph. Grant `Task` deliberately, at the layer that owns the orchestration.

---

## 2. Pass context explicitly in the prompt — assume nothing is inherited

Subagents do not automatically inherit the parent's context and do not share memory between invocations. Anything the worker needs — prior findings, retrieved documents, web search results, customer details — must be written into the spawning prompt.

The failure mode is a worker that "obviously" should know something the coordinator just learned, hallucinating an answer because the actual evidence never crossed the boundary. Treat each Task invocation as if you were briefing a fresh contractor with no access to the project chat.

```text
# Wrong — worker is expected to "just know" what the coordinator found
Task: Synthesise the research findings into a report.

# Right — findings are inlined verbatim
Task: Synthesise the following findings into a report.

<findings>
  <source url="https://...">...verbatim excerpt...</source>
  <source url="https://...">...verbatim excerpt...</source>
</findings>
```

---

## 3. Separate content from metadata with a structured format

When you forward prior agents' outputs (web search results, document analysis, retrieved pages) into the next subagent, use a structured format that keeps content and metadata in separate fields. Source URL, document name, and page number must survive the handoff so the synthesising agent can attribute claims.

Free-form prose loses attribution at every hop. By the time the third subagent writes the report, "according to the FCA guidance" has no anchor back to a URL. A simple tagged or JSON-shaped envelope around each piece of evidence prevents that drift.

```xml
<evidence>
  <item>
    <source_url>https://example.org/policy</source_url>
    <document>Policy Note 2024-03</document>
    <page>7</page>
    <content>...excerpt...</content>
  </item>
</evidence>
```

---

## 4. Emit parallel Task calls in one response, not across turns

To run subagents in parallel, the coordinator must emit multiple `Task` tool calls in a single response. Spreading them across separate turns serialises them: each call waits for the previous to finish before the model gets another turn to issue the next.

This is a property of how the harness schedules tool calls, not of the Task tool itself. The architectural consequence is that any branching plan ("research these five aspects in parallel") must be decided before the coordinator's turn ends — you cannot start one, look at the result, and add a sibling later without losing parallelism.

---

## 5. Design subagent definition files as narrow, tool-restricted roles

Each subagent type is defined by a `.claude/agents/<name>.md` file: YAML frontmatter declaring `description`, `tools`, and `model`, followed by the system prompt in the body. Treat these definitions as roles: a `researcher` that has web search but no write tools, a `synthesiser` that has file write but no search, a `verifier` that has read-only filesystem access.

The narrower the role, the less surface area for the subagent to wander off-task or take destructive action it was never meant to take. Coordinators in turn become easier to reason about because the capabilities of each spawn are visible at the definition site.

```markdown
---
description: Searches the codebase for usage patterns
tools: Read, Grep, Glob
model: sonnet
---

You are a focused code-search agent. ...
```

---

## 6. Write goal-and-criteria prompts, not step-by-step procedures

Coordinator prompts should specify the research goal and the quality criteria the output must meet, not a step-by-step procedure for the subagent to follow. Procedural prompts ("first call tool X, then read file Y, then ...") collapse the subagent's ability to adapt when reality doesn't match the prescribed path.

State what "done" looks like and what the artefact must contain. Let the subagent choose the route. This is also what makes the same subagent definition reusable across different parent tasks — the role description is stable, the goal varies per invocation.

```text
# Wrong — procedural, brittle
Run a web search for "X". Read the first three results. Summarise.

# Right — goal-and-criteria
Produce a 5-bullet briefing on X for a non-specialist reader.
Each bullet must cite a source URL. Flag any conflicts between sources.
```

---

## 7. Use session forking to explore divergent approaches from a shared baseline

Fork-based session management lets several explorations branch from a single completed analysis without redoing the upstream work. The shared prefix — context loading, document retrieval, initial framing — runs once; each fork then takes the conversation in a different direction.

Reach for forking when you want to compare strategies ("conservative refactor vs. rewrite") against an identical starting state, or when an A/B style evaluation needs to factor out variation in the setup. Without forking, each branch repeats the baseline work and risks drifting on details that should be held constant.
