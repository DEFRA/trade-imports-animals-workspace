# Multi-Step Workflows: Enforcement and Handoff

This file covers how to sequence multi-step agent workflows when ordering and prerequisites matter — when a downstream tool must not run until an upstream tool has succeeded, and when a mid-process escalation must hand the conversation off cleanly to another agent or a human. Consult it when designing skills, tool surfaces, or agent loops that involve verification gates, parallel sub-investigations, or escalation paths.

The central distinction is between **programmatic enforcement** (code-level gates that cannot be bypassed) and **prompt-based guidance** (instructions in the system prompt that the model is asked to follow). Both have a place; mixing them up is where deterministic-compliance bugs come from.

---

## 1. Use programmatic gates, not prompt instructions, when compliance must be deterministic

Prompt instructions have a non-zero failure rate. If the workflow involves identity verification before a financial operation, regulatory checks, or any step where one missed call is one breach too many, the ordering must be enforced in code — typically via a hook or a prerequisite gate that refuses to dispatch the downstream tool until the upstream tool has returned a valid result.

Reserve prompt-based ordering for cases where probabilistic compliance is acceptable: style preferences, suggested investigation order, or which tool to try first when multiple would work.

```
Wrong — relies on the prompt
  System prompt: "Always call get_customer before process_refund."

Right — programmatic gate
  process_refund's dispatcher checks the conversation state for a
  verified customer ID from get_customer; if absent, the tool call
  is blocked before it reaches the model's tool-execution layer.
```

The test is simple: if a single failure of this ordering would be unacceptable, gate it in code.

---

## 2. Block downstream tools on the result of their prerequisites, not on the model's claim

A prerequisite gate that checks "did the model emit a get_customer call earlier in the turn?" is not the same as "did get_customer return a verified customer ID?" The first is easy to fool with hallucinated tool calls or stale conversation history; the second pins the gate to an observable fact.

Tie the gate to the **returned value** of the prerequisite tool. For `process_refund`, the gate should look for a verified customer ID in the workflow state — produced by a real `get_customer` response — not for the textual presence of a `get_customer` invocation.

This also makes the gate composable: any future workflow that produces a verified customer ID through a different path (admin override, batch import, escalation tool) can satisfy the same gate without special-casing.

---

## 3. Decompose multi-concern requests into parallel sub-investigations over shared context

When a customer raises several distinct issues in a single message ("my refund didn't arrive AND my account is locked AND I never got the confirmation email"), do not attempt to answer linearly. Decompose the request into distinct items, investigate each in parallel using the shared conversation context, then synthesise a single unified resolution at the end.

Parallel investigation reduces latency and avoids the failure mode where the agent fixates on the first issue and never returns to the others. The shared-context bit matters: each sub-investigation must see the same customer identity, the same account history, and the same earlier turns — otherwise the synthesis step has to reconcile inconsistent views.

The synthesis is the user-facing output. Internal sub-investigation summaries should feed the synthesis, not appear verbatim in the reply.

---

## 4. Produce structured handoff summaries when escalating mid-process

When the workflow escalates — to a human agent, to a different team, to a higher-privilege tool — the receiving party does not have access to the conversation transcript. They have the summary the agent hands them. A free-form paragraph is not enough; the handoff must carry the specific facts the next responder needs to act.

At minimum, a handoff summary includes:

- **Customer ID** (or the canonical identifier for the entity in scope)
- **Root cause analysis** — what the agent has already established about what went wrong
- **Recommended action** — what the agent thinks should happen next, and why
- **Relevant amounts, references, or thresholds** — e.g. refund amount, policy limit, ticket reference

```
Wrong — narrative handoff
  "Customer is upset about a refund. Please help."

Right — structured handoff
  Customer ID:        C-48201
  Root cause:         Refund of £612 exceeds the £500 auto-approval
                      threshold; customer has been waiting 9 days.
  Recommended action: Approve refund manually; policy permits given
                      the order date and customer tier.
  References:         Order ORD-99213, Refund REF-44017
```

The structure is what makes the handoff actionable without re-reading the transcript.

---

## 5. Match the enforcement mechanism to the failure cost

Not every step needs a hook. The decision is a cost-of-failure judgement:

| Failure cost | Mechanism |
|--------------|-----------|
| Regulatory / financial / safety — one breach is unacceptable | Programmatic gate (hook, prerequisite check) |
| Operational quality — occasional misorder is annoying but recoverable | Prompt-based guidance |
| Stylistic / preference — model judgement is fine | No enforcement; let the model decide |

Over-gating produces a brittle workflow where the model cannot recover from edge cases the gate didn't anticipate. Under-gating produces silent compliance failures. The same workflow will typically have steps in all three tiers — gate the financial step, prompt the investigation order, leave the phrasing to the model.
