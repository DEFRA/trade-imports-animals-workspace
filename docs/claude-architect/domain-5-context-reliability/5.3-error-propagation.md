# Error propagation across multi-agent systems

This file covers how subagents should report failures to a coordinating agent so that the coordinator can recover intelligently. Consult it when designing the return contract of a subagent, the error-handling branch of a coordinator, or the synthesis step that aggregates findings across parallel agents. The central principle: a coordinator can only recover well if it receives structured, distinguishable, honest error context — generic statuses, silent suppression, and whole-workflow aborts all destroy the information the coordinator needs.

---

## 1. Return structured error context, not generic status strings

A subagent that fails must return the failure type, what it attempted, any partial results it did gather, and any alternative approaches it considered. Without those four pieces, the coordinator cannot decide whether to retry, route the work elsewhere, degrade gracefully, or abort.

A bare status like `"search unavailable"` hides the very context that would let the coordinator recover. The coordinator does not learn whether the index was timing out (retry later), the query was malformed (rewrite and retry), or the data source has been decommissioned (route to a different agent).

```jsonc
// Wrong — opaque, unactionable
{ "status": "error", "message": "search unavailable" }

// Correct — coordinator can choose its next move
{
  "status": "error",
  "failure_type": "upstream_timeout",
  "attempted_query": "refund flow dependencies in payments-service",
  "partial_results": [
    { "file": "src/refund/RefundController.java", "score": 0.71 }
  ],
  "alternatives": [
    "retry with narrower scope: src/refund/ only",
    "delegate to grep-based subagent if vector search remains unavailable"
  ]
}
```

---

## 2. Distinguish access failures from valid empty results

An access failure (the query could not be evaluated — timeout, auth, missing index) and a valid empty result (the query ran and matched nothing) demand different coordinator decisions. Conflating them is a silent-failure anti-pattern: the coordinator treats "we could not look" as "there is nothing to find" and proceeds with bad data.

Make the distinction explicit in the response shape. A retry is appropriate for an access failure; for an empty result, the coordinator should record the fact and move on.

```jsonc
// Access failure — coordinator should consider retry / fallback
{ "status": "error", "failure_type": "timeout", "attempted_query": "..." }

// Valid empty result — coordinator should record and proceed
{ "status": "ok", "results": [], "queried": "..." }
```

---

## 3. Recover locally for transient failures; only propagate what you cannot resolve

A subagent should absorb retriable problems within its own scope — short network blips, rate-limit backoffs, single-shard timeouts — and only escalate failures it genuinely cannot resolve. Bubbling every transient up to the coordinator turns the coordinator into a retry loop and obscures the real failure signal when something is actually broken.

When the subagent does propagate, it must still include what it attempted (so the coordinator does not duplicate effort) and any partial results it managed to collect before giving up (so downstream synthesis is not starved).

---

## 4. Never silently suppress errors as empty success

Returning `{ "status": "ok", "results": [] }` from a failed query is the most damaging error-handling pattern in a multi-agent system. The coordinator has no signal that anything went wrong; downstream synthesis treats absence-of-evidence as evidence-of-absence; the user receives confidently incomplete output.

If a subagent cannot tell the difference between "the source said nothing" and "I could not ask the source", that is a bug in the subagent's error handling, not a stylistic choice. Fix the subagent to surface the failure type rather than masking it.

---

## 5. Never terminate the whole workflow on a single subagent failure

The mirror-image anti-pattern is treating any subagent failure as fatal and aborting the entire coordination. In a fan-out where one of ten parallel agents fails, the coordinator should preserve the nine successful results, annotate the gap, and decide whether the partial coverage is enough to answer the user — not throw the lot away.

Make this a property of the coordinator, not of each subagent: subagents report what happened; the coordinator decides whether the overall task can still proceed.

---

## 6. Annotate synthesis output with coverage gaps

When the coordinator synthesises findings from multiple subagents, it must mark which conclusions are well-supported and which topic areas have gaps because a source was unavailable. A synthesis that flattens "we checked thoroughly" and "we could not check" into the same prose actively misleads the reader.

Coverage annotation lets the user (and any downstream agent) judge confidence and decide whether to spend more effort closing the gaps.

```markdown
## Findings

- Refund flow is gated by `RefundPolicy.isEligible` — confirmed by code-search agent across 4 call sites.
- Audit-log behaviour: **gap** — audit-log search agent timed out twice;
  the conclusions below do not account for audit-log side effects.
- Webhook retries follow exponential backoff with jitter — confirmed by config-inspection agent.
```
