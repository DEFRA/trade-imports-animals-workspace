# Context Management in Large Codebase Exploration

This file covers how to keep an exploration agent grounded when a single session is too small to hold an entire codebase. Consult it when designing skills, MCP servers, or prompts that drive multi-hour code investigation — tracing flows, mapping dependencies, or auditing a repo — where verbose discovery output would otherwise saturate the main context. The rules below address context degradation, scratchpad persistence, subagent delegation, summary injection, and crash recovery.

---

## 1. Treat extended sessions as a degradation risk, not a free resource

Models in long sessions start giving inconsistent answers and referencing "typical patterns" rather than the specific classes and files discovered earlier in the same session. Once that happens, the agent is hallucinating against its own prior findings.

Design the workflow so the agent never has to rely on context alone to remember a specific finding. Anything the agent will need to cite later — a class name, a call site, a config flag — must be written down outside the conversation. Treat the live context as a working set, not a memory.

---

## 2. Persist key findings to scratchpad files across context boundaries

Have the exploring agent maintain scratchpad files that record key findings, and reference those files for subsequent questions. The scratchpad is the durable record; the context window is just the cursor.

```text
workareas/explore/<topic>/
  findings.md       # bullet list of discovered classes, files, call sites
  open-questions.md # things to investigate next
  decisions.md      # conclusions reached and why
```

When a new question arrives, the agent re-reads the relevant scratchpad section first, then explores. This counteracts the "typical patterns" drift — the model sees the specific names it found earlier, not a generic recollection.

---

## 3. Delegate verbose exploration to subagents; keep the main agent at coordination level

Spawn subagents to investigate specific questions — "find all test files for the refund module," "trace the refund flow dependencies from controller to persistence" — while the main agent preserves high-level coordination. The verbose discovery output (file listings, grep results, AST walks) stays inside the subagent's context and never pollutes the coordinator.

The subagent returns a summary; the coordinator integrates that summary into its model of the system. The coordinator's context fills with structured conclusions, not raw exploration noise.

Right:

```text
main agent: "Subagent A, find every caller of RefundService.process()"
subagent A: <runs grep, opens files, reads call sites, writes findings.md>
subagent A returns: "12 callers across 4 modules; see findings.md §callers"
main agent: <reads the 12-line summary, not the 800 lines of grep output>
```

Wrong: the main agent runs the grep itself, reads every match, and burns half its window on output it will summarise away anyway.

---

## 4. Summarise each exploration phase before spawning the next

Before spawning subagents for phase N+1, summarise the key findings from phase N and inject that summary into the next phase's initial context. Each phase starts with a compact recap of what is already known, not a transcript of how it was discovered.

```text
Phase 1: map the module boundaries → summary: "5 modules: auth, billing, refund, ..."
Phase 2: trace refund flow (input = Phase 1 summary) → summary: "refund touches billing.* and audit.*"
Phase 3: identify test coverage gaps (input = Phase 1+2 summaries)
```

Without the summary step, each phase re-derives prior conclusions from raw output, and inconsistencies creep in as the model paraphrases its earlier self.

---

## 5. Design crash recovery as structured agent state, not chat replay

Replaying a long exploration session from chat history is fragile and slow. Instead, have each agent export its state to a known location as a structured manifest, and have the coordinator load that manifest on resume and inject it into agent prompts.

The manifest is the contract: a JSON or markdown file with a defined schema describing what was explored, what was concluded, what is still open. On resume, the coordinator does not re-read the conversation — it reads the manifest.

```text
workareas/explore/<topic>/
  manifest.json       # { phase, completed_subagents, open_questions, findings_refs }
  agents/
    subagent-a.state.json
    subagent-b.state.json
```

On resume: coordinator reads `manifest.json`, identifies which subagents had completed, and re-spawns only the ones that had not — injecting the manifest summary into each new subagent's initial prompt.

---

## 6. Use /compact to recover context during extended exploration

When context fills with verbose discovery output during a long session, use `/compact` to reduce context usage rather than continuing until the model degrades or truncates. Compaction is a tool to reach for proactively, not a last resort.

Pair it with the scratchpad discipline from rule 2: before compacting, confirm the durable findings are written to disk. Compaction summarises the conversation, but the canonical record of what was discovered should already be in scratchpad files that survive independently.
