# Manage conversation context across long interactions

This file covers how to preserve critical information — transactional facts, issue
data, tool results, subagent findings — across long-running conversations and
multi-turn agent flows. Consult it when designing an agent that accumulates state
over many turns, when wiring a subagent fan-out where downstream context budgets
are tight, or when a session is dropping numbers, dates, or customer-stated
expectations as it grows.

---

## 1. Persist transactional facts in a dedicated case-facts block, not summaries

Progressive summarisation silently degrades the values that matter most:
amounts, percentages, dates, order numbers, statuses, and customer-stated
expectations get rewritten as vague paraphrases. Once a number has been
condensed into "the customer mentioned a refund amount", you can't recover
the original figure from history — and the model will confidently fabricate
a plausible-looking replacement.

Extract these facts into a persistent "case facts" block that is re-included
in every prompt verbatim, sitting **outside** the summarised history. The
summary can compress narrative; the case-facts block must not.

```text
# Wrong — summary swallows the numbers
History summary: "Customer reported an issue with their recent order and
asked for a partial refund. Agent confirmed the order and is investigating."

# Correct — facts pinned separately, summary stays free to compress narrative
Case facts:
  order_id: 8847-A
  order_date: 2026-02-14
  amount_disputed_gbp: 47.50
  customer_expectation: "full refund by Friday"
  status: investigating

History summary: Customer reported damaged goods; agent confirmed order and
is checking warehouse logs.
```

## 2. Layer structured issue data separately for multi-issue sessions

When a single conversation covers multiple distinct issues (multi-order
support, several bug reports, parallel cases), a flat case-facts block
collapses into ambiguity — which order ID went with which dispute? Which
status applies to which line item?

Promote issue data into its own structured context layer, keyed by issue
identifier, alongside the case-facts block. Each issue carries its own
order IDs, amounts, and statuses. Downstream turns can address one issue
without polluting the others.

```text
Issues:
  - id: ISSUE-1
    order_id: 8847-A
    amount_gbp: 47.50
    status: refund_pending
  - id: ISSUE-2
    order_id: 8848-B
    amount_gbp: 12.00
    status: replacement_dispatched
```

## 3. Trim verbose tool outputs to relevant fields before they enter context

Tool results accumulate disproportionately to their relevance. An order-lookup
endpoint may return 40+ fields when only 5 are needed for a returns flow;
multiplied over a session those payloads dominate the token budget and push
genuinely useful content out of the window.

Filter tool outputs at the tool-handler layer, not in the model's head. Keep
only the fields the current task needs, drop the rest before the result is
appended to history. If a later turn needs the trimmed fields, call the tool
again with a more specific projection rather than caching everything "just in
case".

```text
# Wrong — full payload appended to context
{ id, customer, items[20], shipping{...}, billing{...}, audit_log[...],
  promotions[...], tax_breakdown[...], ... 40+ fields ... }

# Correct — projection for a returns task
{ id, status, items: [{sku, qty, return_eligible}], total_gbp }
```

## 4. Counter the "lost in the middle" effect with position-aware structure

Models reliably attend to the beginning and end of long inputs and may omit
findings from the middle. A long aggregated document with the key result
buried at section 7 of 12 will, in practice, be ignored by the model even
though the tokens are technically in context.

Place key findings summaries at the **beginning** of aggregated inputs, and
organise detailed results with explicit section headers so the model can
re-anchor when scanning. Treat the top of the input as prime real estate;
never hide a conclusion in the middle of a long log dump.

```text
# Right — headline up front, details below under headers
## Key findings
- Repo A: 3 high-severity items
- Repo B: clean
- Repo C: 1 medium item, blocked on input

## Detailed results
### Repo A
...
### Repo B
...
### Repo C
...
```

## 5. Treat the conversation history as the agent's working memory — design for what gets pruned

The Claude Code harness keeps the conversation history intact across turns,
but when the conversation grows long it auto-compacts older messages — and
the user can trigger the same effect at any point with `/compact`. Earlier
tool outputs, subagent returns, and intermediate reasoning are not
guaranteed to be visible twenty turns later. Skill authors who assume they
will be are routinely wrong.

Anything an agent needs to remember beyond compaction must live somewhere
durable. The three options, in order of preference: a scratchpad file the
agent re-reads on demand (the canonical pattern for workareas under
`workareas/<skill>/EUDPA-X/`); a subagent's structured return value, which
the orchestrator can re-cite verbatim; or content restated near the end of
the visible window at a handoff point. Free-floating facts in the middle of
a long history are not durable storage.

When designing a slash command or skill that spans many turns, name the
handoff points explicitly — between phases, before spawning a subagent,
before yielding control back to the user — and at each one restate the
load-bearing facts or write them to disk. Treat the visible window as a
cache, not the source of truth; the source of truth is whatever you have
persisted.

---

## 6. Require subagents to return structured outputs with metadata

When subagents return verbose prose and reasoning chains, the synthesising
agent inherits two problems: a blown context budget, and unattributed
findings it can't tie back to source, date, or method. Downstream synthesis
then drifts because the metadata needed to reconcile conflicting results
isn't there.

Mandate that subagents emit structured outputs — key facts, citations,
relevance scores, source locations, dates, and methodological context —
rather than narrative. The orchestrator stays cheap to run, and synthesis
becomes a merge over structured fields instead of a re-read of long
free-text reports.

```text
# Wrong — verbose subagent return
"I looked at the repo and found that there are some issues with the
configuration files, specifically around how the env vars are handled,
which seems to date back to last autumn..."

# Correct — structured return
{
  finding: "env vars hardcoded in compose.yml",
  severity: "medium",
  source: "repos/frontend/compose.yml:42",
  observed_at: "2026-06-08",
  method: "static scan",
  citation: "docker-compose §1"
}
```

---

## 7. Tune upstream agents to fit downstream context budgets

If a downstream agent has a tight context window, an upstream agent that
returns full content plus reasoning will overrun it — even when the
downstream task only needs a handful of facts. The failure surfaces as the
downstream agent silently ignoring late inputs or hallucinating around
truncated ones.

Modify upstream agents to return structured data — key facts, citations,
relevance scores — instead of verbose content and reasoning chains, sized
to what the downstream agent can actually consume. Treat the context budget
as a contract between producer and consumer, not a property of any single
agent in isolation.
