# Multi-Instance and Multi-Pass Review Architectures

This file covers how to architect review of model-generated artefacts (code, plans, analyses) so that bugs and subtle issues are actually surfaced rather than rubber-stamped by the same reasoning that produced them. Consult it when designing review skills, code-review subagent fan-outs, plan-validation steps, or any workflow where one Claude pass critiques another's output. It also applies when a single review pass is being asked to swallow too many files at once.

---

## 1. Use a second independent instance for review, not self-review

A model that has just generated an artefact retains the reasoning context it used to produce it. In the same session it is structurally less likely to challenge its own decisions — the justifications are already cached as part of "what we just decided". Asking it to "now review your work" rarely catches subtle issues for that reason.

Spawn an independent Claude instance with no prior reasoning context to perform the review. The reviewer sees only the artefact and the requirements, not the chain of thought that produced it, so it has to re-derive correctness from scratch.

```
Wrong: one session — generate code, then prompt "now review the above for bugs".
Right: session A generates code; session B (fresh context) reviews only the
       final artefact plus the original requirements.
```

This is the principle behind per-file reviewer subagents and per-item fix subagents in the workspace review skills — each spawns fresh, sees only its slice.

---

## 2. Do not substitute extended thinking for an independent reviewer

Extended thinking blocks live inside the same reasoning context that generated the artefact. They lengthen the reasoning, but they do not break the bias toward defending prior conclusions. Independent review instances are more effective at catching subtle issues than either self-review instructions or extended thinking inside the original session.

Treat thinking blocks and independent review as complementary tools for different problems. Use thinking to help the generator make better decisions; use a fresh instance to challenge those decisions after the fact.

---

## 3. Split large multi-file reviews into per-file passes

A single review pass over many files dilutes attention across the whole set and tends to produce contradictory findings — the same pattern flagged in one file and ignored in another. Per-file local analysis passes keep each pass focused on one artefact and produce more consistent local findings.

Fan out one review subagent per file. Each pass owns a single file, applies the rules, and emits findings scoped to that file. The orchestrator collects results once every per-file pass has completed.

```
Right: N files -> N independent per-file review passes -> aggregate
Wrong: N files -> 1 review pass holding all of them in context
```

---

## 4. Add separate integration passes for cross-file concerns

Per-file passes are blind to cross-file data flow — a contract mismatch between caller and callee, a renamed field that broke a downstream consumer, an enum value handled inconsistently across modules. These need a pass that explicitly looks across files.

Run integration passes after the per-file passes complete, scoped to specific cross-file concerns (data flow, interface compatibility, consistency of conventions). Keep them separate from local review so each pass still has a tight, focused job.

---

## 5. Require self-reported confidence alongside each finding

Without a confidence signal, every finding looks equally important and downstream routing has to treat them all the same. Asking the reviewer to self-report confidence on each finding enables calibrated review routing — high-confidence findings can be auto-actioned or fast-tracked, low-confidence ones queued for a second look or human triage.

Bake the confidence field into the review schema so it is not optional. A finding without a confidence score should not pass validation.

```json
{
  "file": "src/foo.js",
  "line": 42,
  "severity": "high",
  "issue": "Null dereference when input.user is undefined",
  "confidence": "high",
  "fix": "Guard with optional chaining: input.user?.id"
}
```

Use the confidence value to route: high-confidence high-severity to immediate fix, low-confidence to a verification pass or human review.

---

## 6. Make verification passes first-class, not an afterthought

A review architecture that emits only findings — with no mechanism to verify them — assumes every finding is correct. Calibrated routing requires the confidence signal from rule 5 plus an explicit verification stage for findings that fall below the auto-action threshold.

Treat verification as another independent pass with its own fresh instance, scoped to re-examining a specific finding against the artefact. The output is a confirm/refute decision, not a new free-form review.
