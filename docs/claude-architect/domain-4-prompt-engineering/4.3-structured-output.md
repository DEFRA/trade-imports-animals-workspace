# Structured Output via Tool Schemas

This file covers how to extract structured data from Claude reliably, by defining a tool with a JSON schema rather than asking for JSON in plain text. Consult it whenever you need machine-parseable output — extraction pipelines, classifiers, multi-step workflows where one step's output feeds the next, or any place a downstream parser would otherwise crash on malformed JSON.

In Claude Code this pattern shows up two ways: an MCP tool you expose to the agent, or a structured-output tool a subagent is required to call as its final answer (the "return JSON via this tool" pattern enforced by workflow skills).

---

## 1. Define a tool with a JSON schema, not free-text JSON

Asking the model to "respond in JSON" leaves room for missing commas, stray prose, truncated objects, and other syntax errors that crash a strict parser. Defining a tool with a JSON schema as its input parameters is the most reliable approach for guaranteed schema-compliant output — it eliminates JSON syntax errors at the protocol level.

Treat the tool not as an action the model performs but as a typed shape the model is forced to fill in. The "tool" never needs to execute anything; the structured payload is the call's input, which the harness (or the spawning subagent's parent) reads directly.

```json
{
  "name": "extract_invoice",
  "description": "Extract structured fields from an invoice document.",
  "input_schema": {
    "type": "object",
    "properties": {
      "invoice_number": {"type": "string"},
      "total": {"type": "number"},
      "line_items": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "description": {"type": "string"},
            "amount": {"type": "number"}
          }
        }
      }
    },
    "required": ["invoice_number", "total"]
  }
}
```

This is the same shape an MCP server advertises for a tool, and the same shape a subagent's `StructuredOutput` tool uses to constrain its final response.

---

## 2. Decide whether the tool call is optional, required, or forced by name

When you expose multiple tools, the agent's freedom to pick (or skip) the extraction tool silently undermines your guarantee. Three regimes are worth distinguishing:

- **Optional** — the agent may return plain text instead of calling the extraction tool. Acceptable only when text replies are a legitimate outcome.
- **Required, any tool** — the agent must call *some* tool, but may choose which one. Use when you have multiple extraction schemas and the document type is unknown up front (e.g. invoice vs. purchase order vs. credit note).
- **Forced by name** — a specific named tool must run. Use when a particular extraction is mandatory, e.g. as the first step before downstream enrichment, or as the final hand-back from a subagent.

In Claude Code, the cleanest way to enforce "you must end your turn by calling this tool" is to spell it out in the skill prompt or subagent persona ("call `StructuredOutput` exactly once at the end") and to validate the call shape on receipt. Where the agent has many tools and you can't constrain choice in the protocol, prompt-level enforcement plus a hook that rejects empty/text-only completions is the practical equivalent.

If you need guaranteed structured output, leaving the tool optional is the wrong choice.

---

## 3. Remember: schemas eliminate syntax errors, not semantic errors

Strict JSON schemas via tool definitions guarantee the output parses. They do not guarantee the values are correct. Line items may not sum to the stated total. A date may end up in the `customer_name` field. An amount may be off by a decimal place.

Design downstream validation accordingly. Treat schema compliance as the floor, not the ceiling — see [[domain-4-prompt-engineering/4.4-validation-retry-loops]] for the validation and self-correction patterns that catch semantic errors.

---

## 4. Make fields optional (nullable) when the source may not contain them

Marking a field `required` when the source document genuinely lacks the information pressures the model to fabricate a value to satisfy the schema. Hallucinated dates, made-up reference numbers, and invented totals all originate here.

Mark fields nullable / optional whenever the source may legitimately omit them. Reserve `required` for fields that are structurally guaranteed by the document type.

```json
{
  "type": "object",
  "properties": {
    "invoice_number": {"type": "string"},
    "purchase_order_ref": {"type": ["string", "null"]},
    "discount_code": {"type": ["string", "null"]}
  },
  "required": ["invoice_number"]
}
```

---

## 5. Use enums with `unclear` and `other` + detail for extensible categories

Closed enums (`"category": "A" | "B" | "C"`) force the model to pick something even when the document is ambiguous or genuinely belongs to a fourth category. Two patterns avoid this:

- Add `"unclear"` as an enum value for ambiguous cases. This gives the model an honest escape hatch and surfaces ambiguity for human review rather than burying it as a confident wrong answer.
- Add `"other"` paired with a sibling free-text `detail` field for extensible categorisation. The detail field captures novel categories you haven't enumerated yet, and reviewing the `other` cases tells you which new enum values to add.

```json
{
  "category": {
    "type": "string",
    "enum": ["invoice", "credit_note", "purchase_order", "unclear", "other"]
  },
  "category_other_detail": {
    "type": ["string", "null"],
    "description": "Required when category is 'other'."
  }
}
```

---

## 6. Put format-normalisation rules in the prompt, not the schema

A strict schema declares shape. It cannot, on its own, normalise inconsistent source formatting — dates written as `12/03/24`, `2024-03-12`, and `March 12 2024` will all coerce to a `string` field as written.

Pair the schema with explicit normalisation rules in the surrounding prompt (skill instructions, subagent persona, or tool description): "Return all dates in ISO 8601 (YYYY-MM-DD). Return all monetary amounts as numbers in GBP, no currency symbol." The schema guarantees the field is present and typed; the prompt guarantees it is shaped consistently.
