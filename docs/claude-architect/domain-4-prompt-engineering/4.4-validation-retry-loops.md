# Validation, Retry, and Feedback Loops for Extraction

Covers how to build self-correcting extraction pipelines: when to retry with
error feedback, when retries are pointless, and how to design schemas that
surface their own validation failures. Consult this file when designing any
extraction or analysis flow where the model's first response may be wrong in a
recoverable way, and you need to decide between retrying, reshaping the prompt,
or giving up and routing to a human.

---

## 1. Retry with specific validation errors appended to the prompt

A bare retry ("try again") gives the model no new information and tends to
reproduce the same failure. A retry-with-error-feedback request includes the
original document, the failed extraction, and the specific validation errors
that fired — the model can then self-correct against a concrete diff between
what it produced and what the schema or invariants required.

Structure the follow-up request as three parts: the original source document,
the failed extraction verbatim, and the validation errors with enough context
to be actionable.

```text
The previous extraction failed validation:
  - field "calculated_total" (47.50) does not match "stated_total" (52.00)
  - field "currency" missing

Original document:
<...>

Previous extraction:
<...>

Re-extract, correcting these specific errors.
```

---

## 2. Do not retry when the required information is absent from the source

Retries fix format and structural errors. They cannot conjure information that
isn't in the document the model was given. If the required field lives in an
external document, a database, or implicit human knowledge, retrying just
burns tokens and produces increasingly creative hallucinations.

Before retrying, classify the failure:

- **Format / structural error** (wrong nesting, missing field that was present
  in the source, type mismatch) — retry with error feedback is likely to
  succeed.
- **Missing-information error** (field genuinely absent from the source) — do
  not retry. Surface the gap, route to a human, or fetch the missing document
  and re-run.

Mis-classifying a missing-information failure as a format failure is the most
common retry-loop pathology: it costs money and produces fabricated data.

---

## 3. Distinguish semantic validation errors from schema syntax errors

Schema syntax errors — malformed JSON, missing required fields, wrong types —
should be eliminated upstream by using tool use / structured output, which
forces the model's output through a typed schema. If you're catching JSON
parse errors at the application boundary, move the schema into a tool
definition instead.

Semantic validation errors — values that don't sum, fields in the wrong
placement, dates in the wrong order, totals that disagree — survive structured
output because they are about meaning, not syntax. These are the errors that
retry-with-feedback is designed to fix. Keep the two categories separate in
your validation layer so the retry path only sees semantic errors and the
syntax path stays empty.

---

## 4. Design schemas that surface their own inconsistencies

Don't ask the model for a single `total` field and then compare it to your own
calculation. Ask the model for both a `stated_total` (what the document
asserts) and a `calculated_total` (what the line items sum to). A mismatch
between the two is a finding the model itself has surfaced, and you can act on
it without re-reading the document.

The same pattern generalises: add a `conflict_detected` boolean (or similar
flag) any time the source data could be internally inconsistent. The model is
better at noticing the conflict during extraction than your downstream code is
at reconstructing it afterwards.

```json
{
  "line_items": [...],
  "stated_total": 52.00,
  "calculated_total": 47.50,
  "conflict_detected": true,
  "conflict_reason": "stated total exceeds sum of line items by 4.50"
}
```

---

## 5. Track which patterns trigger findings via a `detected_pattern` field

When the same extraction or analysis pipeline is run repeatedly (e.g. a code
scanner producing findings that developers dismiss), you need to know *why*
each finding fired so you can analyse dismissal patterns systematically. Add a
`detected_pattern` field to each structured finding identifying the construct
or rule that triggered it.

Over time, the dismissal rate per `detected_pattern` value tells you which
patterns produce false positives and should be tuned or removed. Without the
field, all you have is a stream of dismissed findings with no way to cluster
them.

```json
{
  "severity": "warning",
  "rule": "unsafe-deserialization",
  "detected_pattern": "ObjectInputStream.readObject without filter",
  "message": "..."
}
```

---

## 6. Make the retry loop bounded and decision-driven, not open-ended

Once you've classified the failure (Rule 2) and the retry produced specific
feedback (Rule 1), allow at most one or two retries before escalating. An
unbounded retry loop on a missing-information failure is the same failure
mode dressed as resilience.

Each retry should carry strictly more information than the previous attempt —
the validation errors from the immediately prior attempt, not a generic
"please try harder". If two passes with progressively more specific feedback
fail, the problem is upstream (missing source, ambiguous schema, model
capability) and another retry will not fix it.
