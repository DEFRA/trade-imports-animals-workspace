# Few-shot Prompting for Ambiguous Scenarios

This file covers when and how to use few-shot examples to drive consistent, well-formatted output from Claude, especially in scenarios where prose instructions alone produce drift. Consult it when you are designing prompts that classify, extract, or judge ambiguous inputs — tool selection, code review findings, document extraction, or any task where you need the model to generalise judgment beyond the cases you spell out explicitly.

---

## 1. Reach for few-shot examples when instructions alone produce inconsistent output

Detailed natural-language instructions are the right starting point, but for consistently formatted, actionable output they are not the most effective lever. Few-shot examples are. If you have iterated on the wording of an instruction and still see drift in format, structure, or judgment, the next move is to add examples — not to write more prose.

The signal that you need examples is variance across runs on the same input shape: different field orders, inconsistent severity wording, or the model sometimes producing output and sometimes refusing to. Examples pin the shape.

---

## 2. Use 2-4 targeted examples that show reasoning for the chosen action

Few-shot examples are most valuable when the scenario is ambiguous — when more than one plausible action exists and the model needs to learn how you want it to discriminate between them. In those cases, each example should not only show the input and the chosen action, but also show the reasoning for why that action was chosen over the plausible alternatives.

Aim for 2-4 examples. Fewer than two and the model can't triangulate the pattern; more than four and you are burning context without proportionate gain. Cover the discriminating axes — pick examples that differ along the dimensions where the model is currently getting it wrong, not examples that all look alike.

---

## 3. Demonstrate the exact output format you want, field by field

Few-shot examples are also the most reliable way to lock in output format. If you want every finding to carry `location`, `issue`, `severity`, and `suggested fix`, show that shape in the examples — in the order, casing, and level of detail you want at inference time.

The model will copy what it sees. If the examples are terse, output will be terse. If the examples include a `severity: medium` field, output will include severity. Spell out in examples what you would otherwise have to defend across a brittle natural-language schema description.

---

## 4. Include examples that distinguish acceptable patterns from genuine issues

For judgment tasks — code review, content moderation, anomaly detection — false positives are the failure mode. The model flags things that look superficially like the target pattern but are actually fine.

The fix is to include examples on both sides of the line: examples where the pattern is present and should be flagged, and examples where a similar-looking pattern is present but is acceptable and should not be flagged. This lets the model generalise the discriminator rather than matching on surface features. The goal is generalisation to novel patterns that fit the rule, not memorisation of the exact cases you wrote down.

---

## 5. Cover varied document structures in extraction examples

Extraction tasks fail when source documents vary in structure — inline citations versus a bibliography section, methodology described in a dedicated section versus embedded inside results, informal versus formal measurement units. The model trained on one structure will produce empty or null values when it encounters another.

Add few-shot examples that show correct extraction across each structural variant you expect to see. One example per variant is usually enough; the point is to demonstrate that the same target field can live in different places, and to show what successful extraction looks like in each case.

---

## 6. Use examples to suppress fabrication on missing fields

When required fields are not actually present in a source document, the model under instruction-only prompting will sometimes invent values to satisfy the schema. Few-shot examples that show correct handling of missing data — extracting what is there, leaving the rest empty or marked as such — reduce this hallucination directly.

This pairs with schema design choices covered in [[domain-4-prompt-engineering/4.3-structured-output]] (nullable fields, `unclear` enum values), but the few-shot examples are what teach the model when to use those affordances. Showing one example where a field is absent and the extraction leaves it unfilled is more effective than instructing "do not invent values" in prose.
