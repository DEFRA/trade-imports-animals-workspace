# Explicit Criteria over Vague Instructions

This file is the reference for designing prompts that demand precision — reviewers, classifiers, extractors, and any other system where false positives erode trust. Consult it whenever you find yourself writing instructions like "be conservative", "only flag high-confidence issues", or "check that X is accurate"; those phrasings are the failure mode this file exists to prevent. Use it alongside the few-shot file in this domain when designing the full prompt.

---

## 1. Replace vague checks with categorical criteria

Vague instructions like "check that comments are accurate" leave the model to invent its own definition of accuracy, which produces noisy, inconsistent findings. A categorical criterion names the exact condition under which the model should fire.

Compare the two framings:

```text
# Wrong — vague, invites every minor mismatch
Check that comments are accurate.

# Correct — categorical, fires only on a defined contradiction
Flag comments only when claimed behaviour contradicts actual code behaviour.
```

The second form gives the model a binary test it can apply: does the comment's claim contradict what the code does? Stylistic drift, outdated phrasing, and aspirational TODOs all fall outside the criterion and are correctly skipped.

---

## 2. Do not rely on confidence-based filtering to fix precision

Instructions like "be conservative" or "only report high-confidence findings" do not improve precision in any measurable way. The model has no calibrated notion of "high confidence" independent of the criteria you give it, so these phrases collapse to "report whatever you would have reported anyway".

Replace confidence hedging with explicit categorical criteria for what is in-scope versus out-of-scope:

- Define which categories to report (bugs, security issues) and which to skip (minor style, local patterns).
- Tie each category to an observable property of the code, not to the model's self-assessed certainty.
- If a category genuinely cannot be expressed categorically, that is a signal to revisit the prompt — not to fall back on "be careful".

---

## 3. Protect developer trust by removing high false-positive categories

False positive rates compound across categories. A reviewer that is excellent at finding null-dereference bugs but noisy on naming conventions will be ignored on both, because developers learn to dismiss its output wholesale.

When a category produces too many false positives:

1. Temporarily disable that category in the prompt.
2. Continue shipping the categories that are accurate so developer trust is preserved.
3. Iterate on the disabled category's criteria offline until precision is acceptable, then re-enable it.

Keeping a noisy category live "until it improves" is the wrong trade-off — the cost is paid against every other category in the same prompt.

---

## 4. Specify which issues to report and which to skip

Write review criteria as two explicit lists: things that must be reported and things that must be skipped. This is more effective than a single in-scope description, because the skip list pre-empts the categories the model would otherwise drift into.

```text
# Report
- Bugs (incorrect logic, off-by-one, null dereference)
- Security issues (injection, auth bypass, secret exposure)

# Skip
- Minor style differences
- Patterns that are local conventions in this repo
```

The skip list is what stops the prompt from regressing into "check that comments are accurate" territory. Without it, the model fills the gap with its own judgement and precision drops.

---

## 5. Define severity levels with concrete code examples

Severity classification is a second axis where vague instructions fail. "High severity means serious bugs" gives the model nothing to anchor on; one run's HIGH is another run's MEDIUM. Define each severity level with a concrete code example of an issue that belongs there.

For each severity level (for example HIGH, MEDIUM, LOW), include:

- A one-sentence categorical description of what qualifies.
- A short code example showing an issue at exactly that level.
- Optionally, a contrasting example of an issue that looks similar but belongs at a different level.

This grounds the classification in observable code patterns rather than in the model's internal sense of seriousness, and produces consistent severity assignments across runs and across reviewers.
